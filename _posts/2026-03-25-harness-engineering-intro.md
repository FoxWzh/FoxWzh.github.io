**Prompt Engineering解决了"能不能用"的问题，Context Engineering解决了"用得好不好"的问题，而Harness Engineering解决的是"能不能在生产环境中可靠运行、持续迭代"的问题。**

---

## 是什么

想象一位经验丰富的赛车手。光有好的驾驶技术（Prompt Engineering）不够，还需要精心调校的赛车（Context Engineering），更需要完整的维修团队、遥测系统、赛道分析和安全规范（Harness Engineering）。

Harness Engineering（测试框架工程，或称"挽具工程"）是围绕AI模型构建的工程基础设施体系。具体而言，它包括：
- **自动化评估框架**：系统性地测试AI输出质量
- **回归测试套件**：防止模型更新或prompt修改破坏已有功能
- **部署管道**：安全地将新版本推向生产环境
- **可观测性系统**：实时监控生产环境中AI的行为

如果说Prompt Engineering是写一段好的指令，那么Harness Engineering是建立一整套保证这段指令（及其所有未来版本）始终正确工作的机器。

---

## 为什么重要：三个产品场景

**场景一：模型供应商悄悄更新版本**
某法律AI产品使用GPT-4，在没有任何通知的情况下，OpenAI对模型进行了细微调整（这在2024年发生过多次）。产品的合同摘要功能开始出现微妙的格式变化，某些关键条款不再被正确标注。没有HE的团队在两周后通过用户投诉发现了这个问题；有HE的团队在模型更新后的1小时内，自动化测试就发出了告警。

**场景二：Prompt优化引发的意外退步**
产品团队为提升某个场景的回答质量修改了system prompt，上线后确实改善了目标场景，但意外地破坏了另外三个边缘场景的表现——而这些场景没有被手动测试覆盖到。没有回归测试的团队，这个问题直到用户大量投诉才被发现。

**场景三：新功能灰度发布的风险控制**
某AI写作助手准备将默认模型从Claude 3.5 Sonnet升级到Claude 3.7 Sonnet，新模型在基准测试上更好，但团队不确定对特定写作风格任务的影响。有HE的团队可以先将10%的流量路由到新模型，对比质量指标，确认无异常后再全量切换。

## PE → CE → HE：三个阶段的演化逻辑

```
阶段一：Prompt Engineering（PE）
时期：AI产品探索期
核心问题："怎么让模型做我想要的事？"
核心工作：写prompt、调参数、试不同模型
局限：靠人工判断质量，无法系统性验证，无法规模化
适用规模：原型/Demo阶段

        ↓ 当产品开始有真实用户

阶段二：Context Engineering（CE）
时期：AI产品成长期
核心问题："怎么给模型提供最合适的信息？"
核心工作：RAG管道、动态context装配、工具调用链
局限：context质量提升了，但没有系统化验证迭代改进是否有效
适用规模：早期商业化阶段（千级DAU）

        ↓ 当产品需要稳定性和持续迭代

阶段三：Harness Engineering（HE）
时期：AI产品成熟期
核心问题："怎么在保持质量的同时安全地持续迭代？"
核心工作：Eval套件、回归测试、金丝雀部署、可观测性
适用规模：规模化商业产品（万级以上DAU）
```

这三个阶段不是互相替代的关系，而是叠加关系。成熟的AI产品需要同时具备良好的PE（好的prompt）、CE（好的context管理）和HE（好的工程保障）。

## Harness的四个核心组件

### 组件一：评估套件（Eval Suite）

Eval Suite是HE的核心，本质上是"AI产品的单元测试"。它由一批精心设计的测试用例组成，每个测试用例包含：
- 输入（用户查询 + context）
- 期望输出（或期望输出的判断标准）
- 评估方法（精确匹配、语义相似度、LLM评判等）

**评估方法的三个层次：**

```
层次1：确定性评估（最快、最便宜）
├── 格式检查：输出是否为合法JSON？
├── 关键词检查：必须包含/排除特定词？
└── 长度检查：回答长度是否在范围内？

层次2：参考答案对比（中等成本）
├── 精确匹配：与标准答案完全一致？
├── ROUGE分数：文本重叠度评估
└── 嵌入相似度：语义相近度评估

层次3：LLM即评判官（LLM-as-Judge，最准确）
├── 用另一个LLM评判输出质量
├── 可设定评判标准（如：有用性、准确性、安全性）
└── 代价：每次评估需要额外的模型调用成本
```

**Eval数据集的构建：**
优秀的eval数据集需要覆盖：核心功能的正常用例（happy path，约60%）、边缘情况（edge cases，约25%）、已知失败案例（回归用例，约15%）。数据集应该随着产品迭代持续维护和扩充。

### 组件二：回归测试（Regression Testing）

回归测试确保每次改动不会破坏已有功能。在传统软件工程中，这已经是标准实践；在AI产品中，它尤为重要，因为改动的来源更多、更隐蔽：

```
触发回归测试的改动类型：

代码层面的改动               AI层面的改动
├── 功能代码变更              ├── Prompt修改（哪怕只改一个词）
├── 依赖库升级                ├── 模型版本切换
└── 基础设施变更              ├── Context装配逻辑调整
                             ├── RAG知识库更新
                             └── 第三方模型API的静默更新
```

AI产品的回归测试必须覆盖AI层面的改动，这是与传统软件回归测试最大的不同之处。

**实践建议：**
- 每次prompt修改后，自动运行完整eval套件
- 设定质量门槛（如：核心指标不得下降超过5%）
- 未通过质量门槛的改动，阻止其进入主分支

### 组件三：金丝雀部署（Canary Deployment）

金丝雀部署源自矿工携带金丝雀入矿井检测毒气的做法——让小部分流量先"踩雷"，保护大部分用户。

**AI产品的金丝雀部署流程：**

```
阶段1：影子模式（Shadow Mode）
├── 新版本处理真实请求，但不返回给用户
├── 对比新旧版本的输出差异
└── 收集初步质量信号，零用户风险

阶段2：小流量灰度（1%-5%）
├── 1%-5%的真实用户看到新版本
├── 监控关键指标：质量分数、错误率、用户反馈
└── 与对照组进行A/B对比

阶段3：扩大灰度（10%-50%）
├── 确认无异常后扩大比例
├── 监控长尾效应（某些边缘场景可能需要更多流量才能暴露）
└── 收集分层数据（不同用户群的表现差异）

阶段4：全量发布
└── 所有流量切换到新版本，保留快速回滚能力
```

### 组件四：可观测性管道（Observability Pipeline）

可观测性是"看见"AI在生产环境中真实行为的能力。没有可观测性，团队只能靠用户投诉得知问题，而不是主动发现问题。

**AI可观测性的四个维度：**

```
┌─────────────────────────────────────────────────────┐
│                AI可观测性四维度                      │
├──────────────┬──────────────────────────────────────┤
│  质量指标    │ 自动化评分（用LLM评判生产流量样本）   │
│              │ 用户反馈信号（点赞/踩、重问率）       │
│              │ 任务完成率                           │
├──────────────┼──────────────────────────────────────┤
│  性能指标    │ 首token延迟（TTFT）                  │
│              │ 完整响应延迟                         │
│              │ Context构建时间                      │
├──────────────┼──────────────────────────────────────┤
│  成本指标    │ 平均每请求token数                    │
│              │ 模型调用成本趋势                     │
│              │ RAG检索成本                          │
├──────────────┼──────────────────────────────────────┤
│  安全指标    │ 越狱/注入攻击检测率                  │
│              │ 有害输出率                           │
│              │ 拒绝率（是否过度拒绝合法请求）        │
└──────────────┴──────────────────────────────────────┘
```

## 为什么HE是AI产品成熟度的标志

从"能用"到"可靠"，这个跨越在传统软件工程中靠的是几十年积累的工程实践（单元测试、持续集成、监控告警）。AI产品正在走同样的路，只是更压缩、更复杂。

**无HE的AI产品面临的典型问题：**
- 模型供应商更新 → 质量悄悄下降 → 用户流失 → 团队仍不知道为什么
- Prompt修改 → 某个细分场景崩了 → 靠用户投诉发现 → 线上事故
- 新功能上线 → 全量部署 → 发现问题 → 紧急回滚 → 已经影响万级用户
- 无法量化"改进"：不知道新版本是真的更好还是更差，只能靠感觉

**有HE的AI产品可以实现：**
- 每次改动都有量化的质量对比，知道改了哪里、好了多少
- 问题在影响真实用户前被自动发现
- 新版本灰度发布，坏了及时止损
- 生产环境质量可监控，趋势可预测

**PM应该推动的工具优先级：**
1. 首先：可观测性工具（Helicone或类似产品）——了解生产中在发生什么
2. 其次：Eval框架（Promptfoo用于初期，Braintrust或LangSmith用于成熟阶段）
3. 然后：将Eval集成进CI/CD（GitHub Actions + Promptfoo是最低成本的起点）
4. 最后：金丝雀部署机制（通常需要工程团队在基础设施层实现）

## 能力边界与局限

**Eval数据集的质量是HE的天花板：** HE的价值取决于eval数据集的质量。如果eval用例不能代表真实用户行为、没有覆盖关键边缘情况，再好的自动化测试也是"在错误的地方精确测量"。eval数据集的构建和维护是一项需要持续投入的工作，不是一次性完成的任务。

**LLM-as-Judge的自身局限：** 用LLM评判LLM输出是HE中的常见模式，但这个方法有其局限：评判LLM本身可能有偏见（如倾向于认为更长的回答更好）；评判标准难以精确定义；对主观性强的任务（如创意写作风格）效果有限。

**HE不能发现未知的未知：** 回归测试只能发现已知的、被eval覆盖的问题。模型的全新失效模式，特别是在新用户群体或新使用场景中出现的，只有通过生产可观测性和人工审查才能发现。

**工程成本不可忽视：** 建立一套完整的HE体系，需要工程时间和持续维护投入。对于初期产品，应该从可观测性和基础eval开始，逐步建立，而非一开始就追求完整覆盖。

## PM视角的五条关键结论

**1. 没有Eval框架的AI产品是"盲飞"——不知道自己飞得好不好、有没有偏航**
"感觉上次改动让效果变好了"不是可靠的产品决策依据。没有量化eval，团队无法知道改动的真实影响，无法安全地迭代，无法向stakeholder证明产品质量在提升。建立eval是AI产品从"玩具"走向"产品"的关键一步。

**2. HE让AI产品迭代速度提升的同时降低风险——这两者不是矛盾的**
很多团队认为严格的测试会拖慢迭代速度。实际上恰恰相反：有了自动化eval和回归测试，工程师修改prompt时可以立即得到质量反馈，不需要漫长的人工测试；金丝雀部署让新功能可以安全快速上线。HE是AI产品高速迭代的保障，而非障碍。

**3. HE是AI PM需要主动推动工程团队建立的基础设施**
工程团队往往优先建设产品功能，而非测试基础设施。AI PM需要理解HE的价值，将其纳入产品路线图，为其分配明确的工程资源，而不是等到出了线上事故再亡羊补牢。

**4. 从可观测性开始，这是HE建设的最小起步**
如果现在什么都没有，第一步不是写eval用例，而是在所有LLM调用前加一层可观测性代理（如Helicone）。先让团队"看见"生产环境在发生什么，再基于真实数据决定eval重点。

**5. Eval数据集应该与产品一起演化，这是持续的产品工作**
每次出现重要的线上问题，应该将其抽象为新的eval用例加入数据集（称为"修复即测试"原则）。每次推出新功能，应该为新功能的关键路径添加eval覆盖。这样，eval数据集随着产品成长，成为越来越有价值的"产品质量文档"。

## 常见误区

**误区1："我们的AI产品太早期，等大了再建HE"**
这是最危险的误区。在没有eval框架的情况下迭代越久，积累的"技术债"越难清偿——早期没有记录的边缘情况、没有量化的基准线，会让后来建立eval变得格外困难。正确的时机是：第一个真实用户来之前，就应该有基础的eval套件和可观测性接入。

**误区2："HE是工程团队的事，PM不需要了解"**
eval数据集的内容（测什么、期望什么输出）是产品决策，不是工程决策。什么叫"好的输出"，是PM最了解的——因为这来自产品目标和用户需求。PM不需要写eval代码，但需要定义eval标准，并推动eval数据集的构建和维护。

**误区3："有了HE，就不会再出现AI质量问题"**
HE降低风险，但不消除风险。它无法捕捉所有未知的失效模式，无法完全模拟真实用户的多样性，无法预见模型供应商未来的所有变化。HE的目标是：让已知问题不重复出现，让新问题更快被发现，让改动的影响可量化——而非承诺"零事故"。

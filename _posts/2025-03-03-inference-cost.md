> 一句话摘要：AI 产品的运营成本 80-90% 来自推理，不是训练——读完你能在技术评审里说清楚成本高在哪、该怎么优化，而不是只会说"换个便宜模型"。

---

## 是什么

训练大模型是一次性的巨大投入，但推理（用模型回答用户问题）是每天每分钟都在发生的持续成本。一个 AI 产品上线后，**推理成本通常占总运营成本的 80-90%**。

推理成本由四个主要因素构成：

1. **FLOPs**：每次推理需要多少计算量
2. **显存（VRAM）**：模型和上下文需要多少 GPU 内存
3. **批处理（Batching）**：如何高效利用 GPU 处理多个请求
4. **推理加速技术**：各种降低延迟和成本的工程手段

---

## 为什么重要

### 场景 1：产品刚上线，API 账单超出预算

按量计费的 API 成本随用户增长线性增加，但很多优化手段能让成本不那么线性——不了解推理成本构成，就不知道从哪里优化。

### 场景 2：用户反映回复太慢

延迟问题不总是"模型太大"造成的，批处理策略、KV Cache 命中率、网络传输都可能是瓶颈。找对原因才能给对方案。

### 场景 3：评估自建 vs 用 API

什么时候自建模型比用 API 便宜？需要理解推理成本结构才能算清楚这笔账。

---

## 核心内容

### 第一块：FLOPs——计算量从哪里来

**FLOPs（浮点运算次数）** 是衡量模型计算量的指标。模型越大、上下文越长，FLOPs 越高。

推理分两个阶段，计算特性完全不同：

**① Prefill 阶段（预填充）**：处理输入 Prompt

- 把用户输入的所有 token 一次性并行处理
- 计算量大，属于**计算密集型**（Compute-bound）
- GPU 的并行算力直接决定这里的速度
- 影响指标：**TTFT（Time to First Token，首字时延）**

**② Decode 阶段（解码）**：逐 token 生成回复

- 每次只生成一个 token，无法并行
- 每生成一个 token 都要读一遍全部模型权重
- 属于**内存带宽密集型**（Memory-bound）
- 影响指标：**TPS（Tokens Per Second，生成速度）**

```
用户发送："帮我写一封道歉信"（10个token）

Prefill：处理这10个token → 极快（并行）
Decode：生成回复（约200个token）→ 一个个生成，是速度瓶颈
```

---

### 第二块：显存——装得下才能跑

GPU 显存（VRAM）需要同时放下：

| 占用来源 | 计算方式 | 示例 |
|---------|---------|------|
| 模型权重 | 约 2GB / 10亿参数（FP16） | 70B 模型 ≈ 140GB |
| KV Cache | 随上下文长度增长 | 100K token ≈ 10-20GB |
| 激活值等 | 约权重的 20% | 70B 模型额外 ≈ 28GB |

**显存不够会怎样**：
- 模型需要切分到多张 GPU，增加通信开销
- KV Cache 被迫压缩或截断，影响长对话质量
- 并发用户数减少

**2024 年的解决方案**：
- **量化（Q4/Q8）**：直接减小模型权重体积
- **PagedAttention**（vLLM 提出）：像操作系统管理内存一样管理 KV Cache，减少碎片化，同等显存能支持 2-4x 更多并发
- **MLA 架构**（DeepSeek）：从架构层压缩 KV Cache 体积

---

### 第三块：批处理——GPU 利用率的关键

GPU 擅长并行计算，但如果每次只处理一个用户的请求，大量算力就浪费了。**批处理（Batching）** 是把多个请求打包一起处理，提高 GPU 利用率。

**静态批处理（早期方式）**：
- 等凑够 N 个请求再一起处理
- 问题：早到的请求要等后来的请求，增加排队延迟

**连续批处理（Continuous Batching，2024 主流）**：
- 一个请求生成完，立刻把新请求插入批次
- GPU 几乎没有空闲时间
- 吞吐量提升 **5-10 倍**，延迟基本不变

```
静态批处理：
  [请求1 -----完成]
  [请求2 ---------完成]   ← 必须等请求1完成才能处理
  
连续批处理：
  [请求1 --][请求3 ----]  ← 请求1完成后立刻插入请求3
  [请求2 -------]
```

---

### 第四块：推理加速技术

| 技术 | 原理 | 效果 | PM 关注点 |
|------|------|------|---------|
| **FlashAttention** | 优化 GPU 内存访问模式，减少 IO | 长上下文推理速度提升 2-4x | 基本已成标配，确认框架是否支持 |
| **推测解码（Speculative Decoding）** | 用小模型预测多个 token，大模型验证 | 生成速度提升 2-3x | 需要额外显存跑小模型 |
| **KV Cache 压缩** | 丢弃不重要的历史 token | 减少显存占用，支持更长上下文 | 可能影响长对话准确率 |
| **模型并行** | 把模型切分到多张 GPU | 支持超大模型 | 增加 GPU 间通信延迟 |
| **量化推理（FP8/INT4）** | 低精度计算 | 速度 + 省显存 | 轻微精度损失 |

---

### 推理成本的关键指标

作为 PM，需要追踪这四个核心指标：

| 指标 | 含义 | 参考目标 |
|------|------|---------|
| **TTFT（首字时延）** | 用户发送后多久看到第一个字 | < 500ms（流畅体验） |
| **TPS（生成速度）** | 每秒输出多少 token | > 30 TPS（阅读不卡顿） |
| **每千 token 成本** | 你的主要运营成本指标 | 视场景，追求持续下降 |
| **GPU 利用率** | 自建时的资源效率 | > 70%（低于 50% 在烧钱） |

---

### 自建 vs 用 API：怎么算账

**API 计费模式**：按 token 付费，弹性强，适合流量波动大或早期阶段。

**自建推理**：固定硬件成本，适合流量稳定且规模大的场景。

粗略估算临界点：

```
一块 H100（约 $2.5/小时）能处理约 4-5 个并发用户（70B 模型）
  → 如果你有 100 个用户同时在线，需要约 20-25 块 H100
  → 每月硬件成本 ≈ $36,000-$45,000

对比 API：
  100 个用户 × 平均每天 50 次对话 × 1000 token/次 × 30天
  = 150M token/月
  @ $5/1M token（中端模型）= $750/月
  
结论：早期用 API，月消耗超过 5-10 亿 token 再考虑自建
```

## 能力边界 / 局限

- **优化有极限**：即使用了所有加速技术，推理的物理下限是模型权重的读取速度，受限于 GPU 内存带宽，无法无限加速。
- **延迟和吞吐量是矛盾的**：连续批处理提高了吞吐量，但对单个用户来说，等待其他请求可能让延迟略微上升。
- **推测解码并非总有效**：对于输出高度不可预测的任务（如创意写作），小模型的预测命中率低，反而降低效率。
- **自建运维成本被低估**：GPU 故障、驱动更新、负载均衡、监控报警……自建的隐性成本往往是硬件成本的 30-50%。

---

## PM 视角的关键结论

1. **推理成本是 AI 产品最重要的运营变量**，应该和 DAU、留存率一样被列入产品核心指标看板。

2. **优化推理成本的优先级**：量化 > 模型降级（路由） > 批处理优化 > 自建基础设施。前两步不需要工程改动，直接换配置就能降成本。

3. **TTFT 是用户体验的核心**，TPS 是次要的。用户能接受"等 0.5 秒看到第一个字"，但无法接受"等 5 秒才开始输出"。Prompt Caching 是优化 TTFT 最快的手段。

4. **不要过早自建**。月 token 消耗超过 5 亿以上、流量相对稳定，再认真评估自建。早期自建主要节省的是钱，浪费的是工程时间。

5. **连续批处理是必选项**，不是优化选项。如果你的推理服务没有用 vLLM 或类似框架，GPU 利用率可能只有 10-20%，在大量浪费算力。

## 常见误区

**误区 1：换更便宜的模型是降成本的唯一办法**

换模型是最直接的手段，但优化批处理策略、开启量化、改善 Prompt 结构降低 token 消耗，通常不需要换模型就能降成本 50% 以上。

**误区 2：显存越多，速度越快**

显存决定的是"能跑多大的模型"和"能支持多少并发"，不直接决定单次推理的速度。单次推理速度主要取决于计算带宽和批处理策略。

**误区 3：自建一定比 API 便宜**

只有在规模足够大、流量足够稳定的情况下，自建才比 API 便宜。大多数中早期产品用 API 的总成本（含工程人力）远低于自建。

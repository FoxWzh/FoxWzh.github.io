**每一个AI产品的背后，都有一套对抗速率限制、控制成本、优化体验的工程机制——理解这些机制，是AI PM从"能用"到"好用"的必修课。**

## 这是什么

当你的AI产品调用OpenAI、Anthropic或Google的模型API时，你面对的不是一个无限容量的魔法盒子，而是一套有配额、有约束、有成本结构的工程系统。速率限制决定你的产品能扛多大并发，重试策略决定你在高峰期是否会大面积报错，流式输出决定用户是否感觉"AI在实时打字"，而成本控制则决定你的毛利率能否撑过B轮。

这篇文章不是工程师的API对接教程，而是帮助AI PM看懂这些工程决策背后的产品逻辑。

## 为什么对PM来说这很重要

**场景一：产品上线即崩**

某教育公司做了一个AI作文批改功能，内测反响很好，公开发布后当天下午涌入3000名学生同时提交作文。系统开始大量返回429错误（Too Many Requests），前端展示"AI开小差了"。问题根源：他们没有评估API速率限制和并发需求之间的缺口，也没有实现排队机制。这不是工程师的失误，是产品在立项时没有把API配额纳入产品约束。

**场景二：用户等待焦虑**

某法律咨询AI产品，生成一份合同分析报告需要40-60秒。非流式输出的实现方式是：用户等待60秒后，页面突然刷出2000字。用户调研反馈"感觉系统卡死了"。改成流式输出后，用户开始边读边等，跳出率从67%降至23%。这个工程决策的价值，比任何一次产品迭代都高。

**场景三：成本失控**

某企业SaaS公司给客户提供AI客服功能，定价每月2000元。上线后发现，每个客服坐席的月均API成本达到800元，产品毛利率为-10%。复盘发现：所有请求都路由到GPT-4 Turbo，而其中70%的请求其实是简单的FAQ问答，用GPT-3.5完全够用。多模型路由是救命的工程决策。

## 核心内容

### 速率限制：TPM vs RPM，两种截然不同的瓶颈

速率限制有两个维度，PM必须同时理解：

**RPM（Requests Per Minute）**：每分钟最多发出多少次API请求，不管每次请求多大。限制的是并发请求的频率。

**TPM（Tokens Per Minute）**：每分钟最多消耗多少token，包含输入+输出token之和。限制的是数据吞吐量。

两个维度会分别触发限制。举例：如果你的RPM配额是60次/分钟，每次请求1000 token，那理论上你有60,000 TPM的吞吐。但如果你的TPM配额只有40,000，那你实际每分钟只能发出40次请求（假设每次均值1000 token）。

对于内容生成类产品（每次请求输出很长），TPM往往先达到上限。对于高并发短问答场景，RPM先触顶。

下表是2025年初各主要平台Tier制度的估算对比（实际数值随时间更新，仅供量级参考）：

**PM的核实建议**：在产品立项时，就要估算DAU×每用户日均请求次数×平均token长度，和当前配额对比，提前3个月申请Tier升级（审核有周期）。

### 429错误与指数退避：别让重试把系统压垮

当请求超出速率限制，API会返回HTTP 429错误。如果程序直接立即重试，会形成"重试风暴"——大量请求同时撞限制，同时重试，反而让问题更严重。

正确的做法是**指数退避（Exponential Backoff）**：每次失败后等待时间翻倍。

```
第1次失败 → 等待 1秒 → 重试
第2次失败 → 等待 2秒 → 重试
第3次失败 → 等待 4秒 → 重试
第4次失败 → 等待 8秒 → 放弃，返回错误给用户
```

但还有一个关键改进：**抖动（Jitter）**。如果100个请求同时遇到429，都等待2秒后再重试，2秒后又同时撞限制。加上随机抖动（比如等待时间在1.5-2.5秒内随机），可以把重试请求打散，避免同步竞争。

这对PM的意义：系统在高峰期出现少量429是正常的，关键是重试机制是否优雅。如果工程团队没有实现指数退避+抖动，高峰期的429会雪崩式扩散。

### 流式输出（Streaming/SSE）：首字延迟是第一用户体验

**非流式**：等待模型生成完整回复（可能30-60秒），一次性返回给前端。

**流式**：模型每生成一个token，立即推送给前端（Server-Sent Events协议），用户看到文字逐字出现，像有人在实时打字。

关键指标是**TTFT（Time To First Token，首字延迟）**：用户提交请求到看到第一个字的时间。流式输出让TTFT通常降至1-3秒，即使完整回复需要30秒，用户体验是"AI在思考然后开始打字"，而非"系统卡死了"。

研究表明，当等待超过8秒没有任何反馈，用户认为系统故障的概率超过50%。流式输出是解决长生成任务用户体验的根本方案。

对PM的意义：流式输出应该是新AI产品的默认选择，非流式只在特定场景下才合理（如批处理任务、后台异步生成）。

### 超时设置的取舍

API请求超时分两类：

- **连接超时**：等待建立连接的最长时间，通常设5-10秒
- **读取超时**：收到第一个字节后等待完整响应的时间

对于流式输出，读取超时需要设得足够长（60-120秒），否则在模型还在生成时，连接会被服务端断掉。

对于非流式长输出任务，超时设置是风险决策：设太短会导致大量任务失败，设太长会让失败的请求占用连接池资源。

**PM需要关注的产品信号**：如果后台日志显示大量超时错误，要区分是"请求太慢"（模型延迟问题）还是"超时设置太激进"（工程配置问题），两者解决方案完全不同。

### 多模型路由（LLM Router）：成熟AI产品的标配

不是所有任务都需要最强的模型。LLM路由的核心思想是：按任务复杂度自动选择最合适的模型。

```
用户请求
    │
    ▼
[复杂度分类器]
    │
    ├── 简单任务（FAQ、格式转换、简单分类）
    │       └──► GPT-3.5 / Claude Haiku  成本: ~$0.001
    │
    ├── 中等任务（摘要、改写、中等推理）
    │       └──► GPT-4o-mini / Claude Sonnet  成本: ~$0.01
    │
    └── 复杂任务（多步骤推理、代码生成、专业分析）
            └──► GPT-4o / Claude 3.5 Sonnet  成本: ~$0.10
```

Martian、RouteLLM等工具已经提供开箱即用的路由方案。一些大型AI产品团队也会自建分类器。典型的成本优化效果：70%请求走小模型，整体API成本降低50-60%，而用户感知质量下降不超过5%。

### 成本控制四法

**方法一：模型降级**
上文提到的路由策略。关键是要建立质量基准线，确保降级后的质量仍满足场景需求。

**方法二：Prompt压缩**
对于有历史对话上下文的场景，随着对话轮次增加，context越来越长，成本线性增加。Prompt压缩技术（如LLMLingua）可以在语义损失极小的情况下，将context压缩30-50%。对于长对话场景，这是直接的成本削减。

**方法三：语义缓存**
传统缓存依赖完全相同的请求。语义缓存识别"语义相近"的请求（如"帮我写封请假邮件"和"写一封请假的邮件"），返回缓存结果。对于高重复率的企业知识库问答场景，缓存命中率可以达到20-40%，直接减少20-40%的API调用成本。GPTCache等工具提供现成实现。

**方法四：批处理（Batch API）**
对于不需要实时响应的任务（如夜间批量生成报告、内容审核、数据处理），使用Batch API可以获得50%的价格折扣（OpenAI Batch API的定价），代价是延迟从秒级变为小时级。

## 能力边界与局限

- 速率限制是平台商的商业策略，也是基础设施保护机制，无法绕过。企业级配额通常需要提前谈判而非自动升级。
- 流式输出在某些场景不适用：需要对完整输出做后处理再展示（如结构化JSON解析）、需要审核后才能显示的内容。
- LLM路由的分类器本身有成本和错误率，如果分类错误，把复杂任务路由到弱模型，反而影响用户体验。路由器需要持续维护和校准。
- 语义缓存在创意写作、个性化内容等场景不适用——用户期望每次都得到不同结果。

## PM视角关键结论

1. **速率限制是AI产品的隐性瓶颈**：产品规划阶段就要估算API配额需求，不要等到上线后才发现卡在Tier 1的50 RPM。提前3个月申请配额升级。

2. **流式输出应该是默认选择**：TTFT是AI产品最重要的UX指标之一，任何生成超过3秒的场景都应该使用流式输出。这是成本为零的用户体验提升。

3. **LLM路由是成熟AI产品的标配**：单一模型策略是过渡期方案。随着产品规模增长，多模型路由是维持毛利率的必要工程投资。

4. **429错误不等于系统故障**：PM在看监控指标时，要区分"偶发429（正常峰值）"和"持续429（配额不足）"，前者靠重试机制处理，后者需要升级配额或优化路由。

5. **成本控制是产品策略的一部分**：API成本不是纯工程问题。哪些任务可以用弱模型、哪些场景可以缓存、哪些任务可以异步处理——这些都是PM和工程师共同的决策。

## 常见误区

**误区一：速率限制只影响高并发产品**

即使是小产品，TPM限制也可能在单用户使用长上下文任务时被触发。一个用户上传10页PDF做分析，可能单次请求就消耗40,000 token，直接触发Tier 1的TPM上限。速率限制的影响跟用户规模无关，跟任务类型有关。

**误区二：流式输出只是前端的事**

流式输出需要后端正确实现SSE协议，CDN/反向代理不能做response buffer（否则会把流式变成非流式），前端需要正确处理chunk数据。这是全栈协作的工程改造，不是"前端改改就好了"。

**误区三：语义缓存适合所有场景**

语义缓存最适合"输入高度相似、输出期望一致"的场景。对于个性化推荐、创意生成、基于用户历史的回复，缓存会让用户得到不相关甚至错误的响应。PM在推动缓存优化时，要明确划定哪些请求类型进缓存池、哪些不进。

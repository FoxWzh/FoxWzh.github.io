> 一句话摘要：这三个概念决定了你的 AI 产品能有多聪明、多快、多贵——读完你能在需求评审里说清楚为什么"加长上下文"不是免费的。

---

## 是什么

### Token：AI 消耗的"燃料单位"

大模型不像人一样直接读字，而是把文本切成一小块一小块的"碎片"来处理，每一块就是一个 **Token**。

- 英文里，1 个 token ≈ 0.75 个单词
- 中文里，1 个汉字通常对应 1-2 个 token
- 100 万 token ≈ 75 万个汉字 ≈ 大概 30 本普通小说

你每次调用 API，账单上计费的就是 token 数量。输入多少 token、输出多少 token，分别计价。

### Context Window：AI 的"工作桌面"

模型在回答你的时候，只能"看见"放在当前窗口里的内容——这个窗口就叫 **Context Window（上下文窗口）**。

想象成一张桌子：桌子越大，你能同时摊开的材料越多。2025 年主流模型的桌子已经很大了：

| 模型 | Context Window 大小 |
|------|---------------------|
| Gemini 2.5 Pro | 100 万 token |
| Claude 3.7 Sonnet | 20 万 token |
| GPT-4o | 12.8 万 token |
| DeepSeek V3 | 12.8 万 token |

但桌子大不等于什么都能放——这里有个关键陷阱，后面会说。

### KV Cache：AI 的"速记草稿本"

大模型生成文字是一个字一个字蹦出来的（这叫"自回归生成"）。如果没有任何优化，每生成一个新字，它都得把前面所有内容重新算一遍——这会慢得无法商用。

**KV Cache** 就是解决这个问题的：把之前算过的中间结果（Key 和 Value 矩阵）存起来，下次直接复用，不用重算。

```
没有 KV Cache：
  生成第 1000 个字 → 重算前 999 个字的注意力 → 慢

有 KV Cache：
  生成第 1000 个字 → 直接读缓存 → 快 10 倍以上
```

---

## 为什么重要

### 场景 1：用户抱怨"越聊越慢"

对话越来越长，如果 KV Cache 没有正确管理，每轮回复都要重新处理整段历史，延迟会指数级增长。这是很多 AI 产品在长对话场景下体验崩坏的直接原因。

### 场景 2："支持上传长文档"功能要花多少钱？

用户上传一份 10 万字的合同，模型读完需要消耗大量 token，而且每次追问都要把这 10 万字重新送进去——除非你用了 **Prompt Caching**（提示词缓存）。开启缓存后，同一份文档只需计算一次，后续查询成本可以降低 **50%-90%**。

### 场景 3：为什么"百万 token 窗口"不等于"百万 token 可用"

这就是著名的 **Lost in the Middle** 问题：模型对窗口头部和尾部的内容记得最清楚，中间段落容易被"忘掉"。所以即使窗口有 100 万 token，真正能被有效利用的可能只有 30-50%。

---

## 核心内容

### Token 的三种计费模式（2025 年现状）

现在 API 计费已经不是简单的"输入+输出"，而是：

| Token 类型 | 说明 | 价格（相对） |
|------------|------|-------------|
| 输入 Token | 你发给模型的内容 | 正常价 |
| 输出 Token | 模型生成的内容 | 通常是输入的 3-5 倍 |
| 缓存命中 Token | 之前缓存过的输入，命中后直接复用 | 原价的 10%-20% |

**PM 的关键动作**：把系统提示词（System Prompt）和固定知识库设计成可缓存的格式，让缓存命中率最大化。

### Context Window 的真实有效边界

业内有个说法叫 **MECW（Maximum Effective Context Window，最大有效上下文窗口）**，意思是：

> 模型官方支持 100 万 token，但实际上在超过 N 万 token 之后，回答质量开始明显下降。

不同模型的 MECW 差异很大。你不能只看官方参数，得自己测或者查第三方评测（比如 RULER、NIAH 测试）。

### KV Cache 吃的是显存，显存决定并发上限

100 万 token 的 KV Cache 大约需要 **15GB GPU 显存**。这意味着：

- 如果每个用户都用满长窗口，服务器并发数会大幅降低
- 推理成本不是线性的，长窗口场景下边际成本急剧上升

这就是为什么很多产品虽然"支持长上下文"，但要么限速、要么单独收费。

### 2025 年的新进展：MLA 架构大幅压缩缓存体积

DeepSeek 推广的 **MLA（Multi-head Latent Attention）** 架构，通过压缩 KV Cache 的存储形式，让同等显存能支撑多出数倍的并发用户。这是 2025 年国产模型能以极低成本提供长上下文服务的关键技术原因之一。

---

## 能力边界 / 局限

- **KV Cache 不是持久记忆**：每次会话结束，缓存就清了。下次开新对话，一切重来。
- **长窗口 ≠ 高准确率**：塞入大量内容后，模型的注意力会被分散，"针对关键信息的检索能力"会下降。
- **Prompt Caching 有条件**：缓存命中要求输入的前缀完全一致，任何一个字不同都会 cache miss。System Prompt 频繁改动会让缓存完全失效。
- **不同 API 平台支持情况不同**：不是所有模型服务商都开放了 Prompt Caching，使用前需确认。

---

## PM 视角的关键结论

1. **Token 是成本单位，也是体验单位**。输出 token 贵于输入 token，推理模型的"思考 token"更贵，选型前要算清楚边际成本。

2. **Context Window 的有效容量远小于标称容量**。产品设计时要留余量，不要把窗口塞满，保持 60-70% 使用率效果最好。

3. **Prompt Caching 是降成本的最快抓手**。只要系统提示词或知识库稳定，开启缓存能让 API 账单直接砍半。

4. **长上下文 vs RAG 不是非此即彼**。2025 年主流做法是混合架构：核心背景放长窗口，海量文档走 RAG，两者结合才是最优解。

5. **KV Cache 消耗决定并发能力**。要支持大量用户同时进行长对话，需要和技术团队认真规划显存资源，这不是纯软件问题。

---

## 常见误区

**误区 1：窗口越大越好，直接选最大的就行**

实际上窗口越大，单次推理成本越高，延迟也越长。大窗口适合"一次性处理大文档"的场景，不适合高频、实时对话。

**误区 2：KV Cache 就是数据库缓存，改动一点内容缓存还在**

KV Cache 是严格的前缀匹配。System Prompt 改了一个标点，缓存就全部失效，得重新计算整个前缀。

**误区 3：支持 100 万 token 就能直接塞进去 100 万字**

实际上超过有效边界的内容会被模型"遗忘"或注意力权重极低，而你仍然要为这些 token 付费。

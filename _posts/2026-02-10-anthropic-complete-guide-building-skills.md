## 原文核心内容

### 第一章：基础概念

**Skill 是什么**

Skill 就是一个文件夹，里面的核心是 `SKILL.md`——用 Markdown 写的指令集，加上 YAML frontmatter 元数据。它不是代码，是"给 Claude 的知识"。

```
your-skill-name/
├── SKILL.md          # 必须，主指令文件
├── scripts/          # 可选，可执行脚本（Python、Bash 等）
├── references/       # 可选，参考文档，按需加载
└── assets/           # 可选，模板、字体、图标
```

重要规则：
- 文件名必须是 `SKILL.md`（大小写严格，不接受 `skill.md` 或 `SKILL.MD`）
- 文件夹名必须是 kebab-case（`notion-project-setup` ✅，`NotionProject` ❌）
- 文件夹内**不能有** `README.md`（放到仓库根目录给人类读，Skill 文件夹只给 Claude 读）

**三级渐进式披露（Progressive Disclosure）**

这是 Skill 整个架构的核心设计哲学：

| 层级 | 内容 | 何时加载 | 设计目的 |
|------|------|---------|---------|
| 第一级：YAML frontmatter | 名称 + 描述 | **始终在系统提示里** | 让 Claude 知道这个 Skill 存在，决定是否激活 |
| 第二级：SKILL.md 正文 | 完整指令和最佳实践 | Claude 判断 Skill 相关时加载 | 提供详细执行指导 |
| 第三级：references/ 文件 | 补充文档、API 参考 | Claude 按需读取 | 避免一次性塞满 context |

逻辑很清晰：frontmatter 永远在，但很小；正文按需加载；详细文档进一步按需读取。这套设计在保证专业能力的同时最小化 token 消耗。

**三个核心设计原则**

1. **渐进式披露**：如上，分层加载，最小化 token 使用
2. **可组合性（Composability）**：Claude 可以同时加载多个 Skill，每个 Skill 设计时不应假设自己是唯一的
3. **可移植性（Portability）**：同一个 Skill 在 Claude.ai、Claude Code、API 三个平台上表现完全一致，一次构建，全平台通用

**Skill 和 MCP 的关系**

原文用了一个厨房比喻，说得很准：

> MCP 是专业厨房——提供工具、原材料、设备（连接到你的服务，让 Claude 能调用真实 API）  
> Skill 是食谱——告诉 Claude 怎么用这些工具，用什么顺序，注意什么

没有 Skill 的 MCP 集成：用户连上了，但不知道怎么用，每次都要从头解释，结果不一致，support ticket 飞起来。有了 Skill：工作流自动激活，最佳实践嵌入每次交互，新用户第一次就能上手。

### 第二章：规划与设计

**从用例出发，不要从工具出发**

原文给了一个很好的用例定义框架：

```
用例名称：Sprint 规划
触发条件：用户说"帮我规划这个 Sprint"或"创建 Sprint 任务"
执行步骤：
  1. 从 Linear 获取当前项目状态（via MCP）
  2. 分析团队速度和容量
  3. 建议任务优先级
  4. 在 Linear 里创建带标签和估时的任务
预期结果：完整规划好的 Sprint，任务已创建
```

在写任何代码或指令之前，先定义 2-3 个具体用例，问清楚：
- 用户想完成什么？
- 需要哪些多步骤流程？
- 需要哪些工具（内置的还是 MCP 的）？
- 哪些领域知识或最佳实践需要嵌入？

**三类常见 Skill 用例**

Anthropic 从早期用户观察中总结出三类最常见的 Skill：

**类别一：文档与资产创建**

用于生成一致、高质量的输出——文档、PPT、设计、代码等。代表：`frontend-design` Skill。

核心技法：
- 把风格指南和品牌标准嵌入 Skill
- 提供输出模板结构
- 在最终交付前加质量检查清单
- 不需要外部工具，只用 Claude 内置能力

**类别二：工作流自动化**

用于需要一致方法论的多步骤流程，通常跨多个 MCP 服务。代表：`skill-creator` Skill。

核心技法：
- 有验证节点的逐步工作流
- 常见结构的模板
- 内置审查和改进建议
- 迭代精炼循环

**类别三：MCP 增强**

为现有 MCP 服务器提供工作流指导。代表：Sentry 的 `sentry-code-review` Skill（自动分析并修复 GitHub PR 中被 Sentry 检测到的 Bug）。

核心技法：
- 顺序协调多个 MCP 调用
- 嵌入领域专业知识
- 提供用户通常需要手动指定的上下文
- 处理常见 MCP 错误

**成功标准要提前定义**

原文承认这还不是精确科学，有点"靠感觉"，但给出了基本框架：

量化指标：
- Skill 对相关查询的触发率达到 90%（测法：跑 10-20 个测试查询，统计自动触发次数）
- 工作流在 X 次工具调用内完成（测法：对比有/无 Skill 的 token 消耗和工具调用次数）
- 每个工作流 0 次 API 调用失败

质化指标：
- 用户不需要手动提示 Claude 下一步做什么
- 工作流在没有用户纠正的情况下完成
- 跨 Session 结果一致（新用户第一次尝试就能成功）

**YAML frontmatter：最重要的部分**

原文用专门的篇幅强调：frontmatter 是 Claude 决定是否加载 Skill 的唯一依据，写不好等于 Skill 不存在。

最简必要格式：
```yaml
---
name: your-skill-name
description: 做什么。当用户询问[具体场景]时使用。
---
```

字段规则：

`name`：
- 只能是 kebab-case
- 不能有空格或大写
- 必须和文件夹名一致

`description`（最关键）：
- 必须同时包含 **What（做什么）** 和 **When（什么时候用）**
- 1024 字符以内
- 不能有 XML 尖括号（安全限制，因为 frontmatter 会注入系统提示）
- 要包含用户实际会说的话

好的 description 长这样：
```yaml
# 具体、可操作
description: 分析 Figma 设计文件并生成开发交付文档。当用户上传
.fig 文件、要求"设计规范"、"组件文档"或"设计到代码的交接"时使用。

# 包含触发短语
description: 管理 Linear 项目工作流，包括 Sprint 规划、任务创建
和状态追踪。当用户提到"Sprint"、"Linear 任务"、"项目规划"或
要求"创建 ticket"时使用。
```

差的 description：
```yaml
# 太模糊，没有触发条件
description: 帮助处理项目。

# 有功能描述，但缺触发条件
description: 创建复杂的多页面文档系统。

# 太技术，用户不会这么说
description: 实现带层级关系的 Project 实体模型。
```

**安全限制**：frontmatter 里不能用 XML 尖括号（`< >`），Skill 名不能包含"claude"或"anthropic"（保留名称）。原因：frontmatter 会出现在 Claude 的系统提示里，恶意内容可能注入指令。

**写指令的最佳实践**

- 具体且可执行，不要模糊
- 包含错误处理（常见报错 → 原因 → 解法）
- 清晰引用打包的 references 资源
- 用渐进式披露：SKILL.md 只放核心指令，详细文档放 references/ 并链接

一个实用技巧：对于关键验证步骤，用脚本而不是语言指令。原文这样说——

> 代码是确定性的；语言解释不是。

如果你的 Skill 需要"在调用 create_project 前验证项目名非空、至少有一个成员"，写一个 `validate.py` 来做这个检查，比在 SKILL.md 里写"请确保验证这些字段"可靠得多。

### 第三章：测试与迭代

**三个层级的测试**

原文把测试分成三档，按质量要求和 Skill 覆盖范围选择：
- **Claude.ai 手动测试**：直接跑查询看行为，迭代快，不需要额外设置
- **Claude Code 脚本测试**：自动化测试用例，跨变更的可重复验证
- **API 编程测试**：构建系统化的评估套件

**一个关键的迭代建议**

> 先在单个挑战性任务上迭代，直到 Claude 成功，再把成功路径提炼成 Skill。

这比一开始就覆盖所有场景高效得多。原因：利用了 Claude 的 in-context learning，能快速得到信号。有了可用的基础，再扩展到多个测试用例。

**测试要覆盖三个维度**

**1. 触发测试**——确保 Skill 在该激活时激活，不该激活时不激活

```
应该触发：
- "帮我在 ProjectHub 里设置一个新工作区"
- "我需要在 ProjectHub 里创建一个项目"
- "为 Q4 规划初始化一个 ProjectHub 项目"

不应该触发：
- "旧金山今天天气怎么样？"
- "帮我写 Python 代码"
- "创建一个电子表格"（除非你的 Skill 专门处理这个）
```

**2. 功能测试**——验证 Skill 产出正确输出

```
测试：用 5 个任务创建项目
给定：项目名"Q4 规划"，5 个任务描述
执行 Skill 工作流后：
  - 项目在 ProjectHub 中创建 ✓
  - 5 个任务以正确属性创建 ✓
  - 所有任务链接到项目 ✓
  - 无 API 错误 ✓
```

**3. 性能对比**——证明 Skill 真的提升了效果

```
不使用 Skill：
  - 用户每次提供指令
  - 15 轮对话来回
  - 3 次 API 调用失败需要重试
  - 消耗 12,000 tokens

使用 Skill：
  - 自动执行工作流
  - 仅 2 个澄清问题
  - 0 次 API 调用失败
  - 消耗 6,000 tokens
```

**迭代信号识别**

触发不足的信号：
- Skill 在应该加载时没有加载
- 用户在手动启用它
- 有"什么时候用这个 Skill"的支持问题

解法：在 description 里加更多细节和关键词，特别是技术术语。

触发过度的信号：
- Skill 在不相关查询上加载
- 用户在禁用它
- 用途模糊

解法：加负向触发条件，更明确地界定范围：
```yaml
description: 高级 CSV 文件数据分析。用于统计建模、回归、聚类。
不用于简单数据探索（改用 data-viz skill）。
```

执行问题的信号：
- 结果不一致
- API 调用失败
- 用户需要纠正输出

解法：改进指令，加错误处理。

### 第四章：分发与共享

**当前分发模式（2026年1月）**

个人用户：
1. 下载 Skill 文件夹
2. 压缩成 zip
3. 上传到 Claude.ai → 设置 → 能力 → Skills
4. 或放到 Claude Code 的 skills 目录

组织级：2025年12月18日发布，管理员可以全工作区统一部署，支持自动更新和集中管理。

**开放标准**

Anthropic 明确把 Skill 定位为开放标准——就像 MCP 一样，Skill 应该跨平台可用，不绑定 Claude。他们已在和生态系统伙伴合作推进这个标准。

**API 使用 Skill**

```
/v1/skills 端点管理 Skill
在 Messages API 请求中通过 container.skills 参数添加 Skill
需要 Code Execution Tool beta（提供 Skill 运行所需的安全环境）
```

**推荐的分发流程**

1. 在 GitHub 开公开仓库，写清楚的 README（给人读），Skill 文件夹里不放 README
2. 在你的 MCP 文档里链接 Skill，说明两者结合的价值
3. 提供快速入门指南

**如何介绍你的 Skill（positioning）**

原文给了很有用的对比：

```
❌ 差的写法（功能视角）：
"ProjectHub Skill 是一个包含 YAML frontmatter 和 Markdown
指令的文件夹，调用我们的 MCP 服务器工具。"

✅ 好的写法（价值视角）：
"ProjectHub Skill 让团队在几秒内完整建好项目工作区——包括
页面、数据库、模板——而不是手动操作 30 分钟。"
```

### 第五章：模式与问题排查

**问题优先 vs 工具优先**

原文用 Home Depot 比喻：你可以带着问题去（"我需要修厨柜"，员工帮你找工具），也可以先挑好工具再问怎么用。

Skill 也有这两种思路：
- **问题优先**：用户描述目标，Skill 协调正确的 MCP 调用序列
- **工具优先**：用户已有 MCP 接入，Skill 教 Claude 最优工作流

大多数 Skill 会偏向其中一种。

**五种常见模式**

**模式一：顺序工作流编排**

用于需要按特定顺序执行多步骤流程的场景。

关键技法：
- 明确的步骤顺序
- 步骤间的依赖关系
- 每个阶段的验证
- 失败时的回滚指令

**模式二：多 MCP 协调**

用于跨多个服务的工作流。

典型案例：设计到开发的交接流程
- Phase 1（Figma MCP）：导出设计资产，生成设计规范
- Phase 2（Drive MCP）：创建项目文件夹，上传资产
- Phase 3（Linear MCP）：创建开发任务，附上资产链接
- Phase 4（Slack MCP）：在 #engineering 频道发布交接摘要

关键技法：
- 清晰的阶段分隔
- MCP 之间的数据传递
- 进入下一阶段前的验证
- 集中式错误处理

**模式三：迭代精炼**

用于输出质量通过迭代改善的场景（如报告生成）。

流程：初稿 → 质量检查（运行验证脚本）→ 精炼循环（处理每个问题 → 重新生成 → 重新验证）→ 最终交付

关键技法：
- 明确的质量标准
- 验证脚本
- 知道什么时候停止迭代

**模式四：上下文感知工具选择**

用于同一目标根据上下文选择不同工具的场景。

示例——智能文件存储决策树：
- 大文件（>10MB）→ 云存储 MCP
- 协作文档 → Notion/Docs MCP
- 代码文件 → GitHub MCP
- 临时文件 → 本地存储

关键技法：
- 清晰的决策标准
- 备选方案
- 向用户解释为什么选了这个存储方式

**模式五：领域专业知识嵌入**

用于 Skill 需要添加工具访问之外的专业知识的场景。

示例——带合规检查的支付处理：
- 处理前（合规检查）：获取交易详情 → 对照制裁名单 → 验证管辖权 → 评估风险 → 记录合规决策
- 处理中：合规通过 → 调用支付处理 MCP → 处理交易；不通过 → 标记审查 → 创建合规案例
- 处理后：记录所有合规检查，生成审计报告

关键技法：
- 把领域专业知识嵌入执行逻辑
- 行动前的合规检查
- 完整的决策记录
- 清晰的治理流程

**常见报错排查**

*Skill 上传失败*

```
错误："Could not find SKILL.md in uploaded folder"
原因：文件名大小写不对
解法：重命名为 SKILL.md（区分大小写），用 ls -la 确认

错误："Invalid frontmatter"
原因：YAML 格式问题
常见错误：缺少 --- 分隔符、引号未闭合

错误："Invalid skill name"
原因：名称有空格或大写
解法：改成 kebab-case
```

*Skill 不触发*

快速检查 description：
- 是否太通用？（"帮助处理项目"不行）
- 是否包含用户实际会说的触发短语？
- 如果涉及文件类型，是否提到了？

调试技巧：问 Claude "你什么时候会用 [skill 名称] 这个 Skill？"——Claude 会把 description 复述出来，你就知道哪里有问题。

*Skill 触发太频繁*

方法一：加负向触发
```yaml
description: 高级 CSV 数据分析。用于统计建模、回归、聚类。
不用于简单数据探索（改用 data-viz skill）。
```

方法二：更具体地界定范围
```yaml
# 太宽泛
description: 处理文档

# 更具体
description: 处理 PDF 法律文件用于合同审查
```

*MCP 连接问题*

排查顺序：
1. 确认 MCP 服务器已连接（Claude.ai → 设置 → 扩展，应该显示"已连接"）
2. 检查认证（API Key 是否有效，OAuth 是否刷新）
3. 独立测试 MCP（不用 Skill，直接让 Claude 调用 MCP；如果这步失败，是 MCP 问题不是 Skill 问题）
4. 核实工具名称（Skill 里引用的 MCP 工具名是否精确匹配，区分大小写）

*指令没被遵守*

常见原因和解法：
1. 指令太啰嗦 → 保持简洁，用列表，把详细参考移到 references/
2. 关键指令被埋没 → 放到顶部，用 `## 重要` 或 `## 关键` 标题
3. 语言模糊 → 用精确语言，必要时写验证脚本替代语言描述
4. 模型"偷懒" → 加明确激励："请花时间彻底完成"、"质量比速度重要"、"不要跳过验证步骤"（注意：这些在用户提示里比在 SKILL.md 里效果更好）

*上下文过大导致性能下降*

原因：
- SKILL.md 内容太大
- 同时启用的 Skill 太多（超过 20-50 个）
- 所有内容一次性加载，没用好渐进式披露

解法：
- 把详细文档移到 references/，SKILL.md 控制在 5000 词以内
- 评估同时启用的 Skill 数量，考虑按需启用
- 检查是否把不应该在 frontmatter 里的内容放进去了

## 我的解读

### 这份文档在说什么

表面上是一份 Skill 技术手册，本质上是 Anthropic 在告诉你：**AI 能力要真正变成生产力，需要"知识工程化"这一层**。

Skill 的出现解决了 AI 使用中的一个核心问题：同一个任务，不同人问出来的结果天差地别，而且每次都要从头建立上下文。Skill 的答案是——把最佳实践固化，让 AI 对话变成可重复的工程流程。

这和"流程标准化"在传统软件工程里的意义是同构的。区别在于，传统流程标准化是写 SOP、写文档，Skill 是把这些 SOP 变成 AI 可以直接执行的指令集。

### 三个值得认真对待的细节

**1. description 是产品入口，不是技术元数据**

大多数人会把 description 当成"顺手填的说明"。但 Anthropic 几乎用了一整节来强调：description 写得好不好，决定 Skill 有没有用。

背后的道理：Claude 在判断"是否激活这个 Skill"时，看的就是 description。这相当于你的产品在系统提示层面的 UX——你需要预判用户会用什么语言描述需求，再把这些语言映射成触发规则。

这对 PM 的启发是：每个 Skill 的 description 应该像在写产品的 onboarding copy，而不是技术注释。

**2. "代码是确定性的，语言不是"**

原文在讲关键验证时说：与其在 SKILL.md 里写"请验证这些字段"，不如写一个 `validate.py` 来做检查。这一句话浓缩了 AI 产品工程的一个核心判断框架：

- 对于有明确对错的判断（字段是否为空、值是否在范围内）→ 用代码做，确定性高
- 对于需要理解和推理的判断（这个回答是否符合风格）→ 用语言指令，让模型判断

很多团队在设计 AI 工作流时把这两类混在一起，结果要么过度信任模型做确定性判断，要么过度用代码做本该交给模型的推理工作。

**3. 渐进式披露是 token 经济学的产品表现**

为什么不把所有信息都塞进 SKILL.md 正文？因为 frontmatter 始终在系统提示里，太大会消耗大量上下文，影响模型在当前任务上的表现。

这是 LLM 产品特有的"性能 vs 能力"权衡：信息越完整，context 消耗越大，模型在其他地方能用的空间就越少。渐进式披露是这个权衡的解法——分层加载，只在需要时加载。

对 PM 的类比：这和前端的懒加载（lazy loading）是同构的问题。你不会在首屏加载所有资源，你在需要时才加载。Skill 设计同理：什么信息必须随时可见，什么按需加载，需要提前想清楚。

### 当前的真实局限

**分发体验还比较原始**

2026年1月的分发方式：下载 → 压缩 → 手动上传。对非技术用户来说门槛不低。GPTs 的分发体验好很多（一个链接分享）。Anthropic 推出了组织级统一部署，但个人用户的分发体验还有很大提升空间。

**评测方法论还在摸索**

原文诚实地承认了这一点："这些是估算目标，而不是精确阈值。我们正在开发更严格的测量指南。"也就是说，目前判断一个 Skill 好不好，主要还是靠人工感知，没有成熟的量化体系。对于需要向团队汇报 AI 投入 ROI 的 PM，这是一个真实的挑战。

**同时启用 Skill 数量有上限**

文档提到超过 20-50 个 Skill 同时启用会导致性能下降。这意味着 Skill 需要按场景管理，不能无限堆叠。

## 延伸思考

**Skill 生态会像 App Store 一样演化吗？**

Anthropic 把 Skill 定义为"开放标准"，希望它跨平台通用。如果这个方向成立，围绕 Skill 会出现的东西和 App Store 早期很像：通用 Skill 开发者、垂直行业 Skill、Skill 集合包、Skill 质量评测榜。

对于正在做 AI 工具产品的团队：现在是入局这个生态的早期窗口。如果你的产品已经有 MCP 集成，配套发布官方 Skill 是相对低成本的差异化动作，能显著降低用户上手门槛。

**"先在一个任务上迭代到成功"是最被低估的建议**

原文建议在构建 Skill 时，先聚焦在单个挑战性任务上反复迭代直到成功，再提炼成 Skill。

这听起来很朴素，但背后有一个重要的方法论含义：在 in-context 里跑通，比在 SKILL.md 里凭空设计更有效。你在对话里发现了什么让 Claude 成功的关键步骤，再把这些关键步骤结构化到 Skill 里。

这和"先做 MVP 再规模化"的产品思维完全一致——先验证核心价值，再工程化。

**Skill 是知识蒸馏的工具**

从更高的视角看，Skill 做的事情是把人类的隐性知识（怎么做好这件事的经验）转化成显性的、可执行的 AI 指令。这是一种知识蒸馏。

随着 Skill 的使用和迭代，这个知识会越来越精准。组织里越多人用、越多人改进，Skill 就越好。这是一个需要启动成本但有复利的过程——第一版 Skill 的质量决定了这个飞轮能不能转起来。

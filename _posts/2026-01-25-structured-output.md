**结构化输出是 AI 产品工程化的关键一步——从"能用"到"可靠"的差距，很大程度上取决于你是否为下游系统设计了可解析、可验证的输出格式。**

## 是什么

结构化输出（Structured Output）是指要求 AI 模型按照预定义的格式（通常是 JSON、XML 或特定模板）返回结果，而不是自由形式的自然语言文本。

自 2024 年下半年起，OpenAI、Anthropic、Google 等主流模型提供商相继推出了更强的结构化输出能力。OpenAI 在 2024 年 8 月正式发布 Structured Outputs 功能，支持通过 JSON Schema 强制约束输出格式；Anthropic 的 Claude 系列则通过精心设计的 prompt + Tool Use 机制实现类似效果。

结构化输出解决的核心问题是：**让 AI 的输出从"人类可读"变为"机器可解析"**。

## 为什么对 PM 重要

**场景一：工作流自动化**

某 HR SaaS 产品使用 AI 解析简历，提取候选人的工作经历、教育背景、技能标签。如果模型输出的是自然语言段落，下游系统就需要再次解析这段文本，引入二次错误；如果输出的是结构化 JSON，可以直接写入数据库字段，整个流程自动化率从 60% 提升至 95%。

**场景二：多步骤 Agent 系统**

在 Agentic 工作流中，每个步骤的输出往往是下一个步骤的输入。如果中间节点输出自然语言，编排层需要写复杂的解析逻辑，且极易因格式不一致而崩溃。某电商的订单处理 Agent，通过强制每个步骤输出 JSON，将系统稳定性从 78% 提升至 97%。

**场景三：数据分析与报表生成**

一家医疗信息化公司使用 AI 从非结构化病历中提取标准化字段（诊断代码、用药记录、手术信息）。非结构化输出的人工校验时间为每条记录 3-5 分钟；引入结构化输出后，系统可以自动完成 90% 的字段填充，人工只需核对异常项，效率提升 8 倍。

## 核心内容

### 为什么要结构化输出

大语言模型天然倾向于生成流畅的自然语言。当我们把 AI 集成进产品工作流时，这个特性反而成了障碍。

自然语言输出存在三个核心问题：
1. **不可预测的格式**：同样的问题，今天返回"用户年龄是 25 岁"，明天可能返回"年龄：25"或"25"。
2. **难以解析**：代码需要用正则表达式或 NLP 方法提取信息，引入额外的失败点。
3. **版本不稳定**：模型更新后，输出格式可能悄然改变，导致线上解析代码静默失败。

### JSON Mode vs 自然语言输出

| 维度 | 自然语言输出 | JSON Mode |
|------|------------|-----------|
| 可解析性 | 需要二次处理 | 直接反序列化 |
| 格式稳定性 | 低，每次可能不同 | 高，schema 约束 |
| 错误检测 | 难以自动检测 | schema 验证即可发现 |
| 人类可读性 | 高 | 较低 |
| 适用场景 | 最终呈现给用户 | 中间处理、数据存储 |
| Token 消耗 | 相对较少 | 略多（字段名占用 token）|

**实践原则**：凡是需要"程序处理"的输出，用结构化；凡是需要"人阅读"的输出，用自然语言。在同一个 API 调用中，有时可以两者兼顾——先用 JSON 输出结构化数据，再加一个 `summary` 字段给用户展示。

### Schema 设计原则

Schema 的质量直接决定结构化输出的质量。以下是经过实践验证的设计原则：

**原则一：描述字段（description）是关键，不是可选项**

```json
{
  "name": "user_sentiment",
  "type": "string",
  "enum": ["positive", "negative", "neutral"],
  "description": "用户对产品的整体情感倾向。
    正面：表达满意、推荐、赞美的评价；
    负面：表达不满、投诉、退款意图的评价；
    中性：只描述事实、无明显情感倾向的评价。"
}
```

对比没有 description 的版本，有 description 的 schema 输出准确率提升约 15-25%。

**原则二：枚举值优于自由文本**

```json
// 不推荐：自由文本，模型可能返回 "中等" "还行" "一般般"
{
  "quality": "string"
}

// 推荐：枚举值，输出完全可控
{
  "quality": {
    "type": "string",
    "enum": ["excellent", "good", "average", "poor"]
  }
}
```

**原则三：区分必填和可选字段**

非所有字段都能在所有情况下提取到。强制要求所有字段必填会导致模型"编造"缺失字段的值。应该将"有时存在"的字段标记为可选（nullable），并在代码中做 null 处理。

```json
{
  "required": ["order_id", "status", "amount"],
  "properties": {
    "order_id": {"type": "string"},
    "status": {"type": "string", "enum": ["pending", "paid", "cancelled"]},
    "amount": {"type": "number"},
    "discount_code": {"type": ["string", "null"], "description": "优惠券代码，无则为 null"}
  }
}
```

**原则四：嵌套层级控制在 3 层以内**

嵌套超过 3 层的 schema 会显著增加模型的"格式遵循负担"，错误率随层级指数增加。如果业务逻辑需要复杂嵌套，考虑拆分成多次 API 调用。

### Schema 验证库的配合使用

**Python：Pydantic**

Pydantic 是 Python 生态中结构化输出的标准配套工具，与 OpenAI SDK 和 Anthropic SDK 都有良好集成。

```python
from pydantic import BaseModel, Field
from typing import Optional, Literal

class OrderAnalysis(BaseModel):
    order_id: str = Field(description="订单号，格式为 ORD-XXXXXXXX")
    sentiment: Literal["positive", "negative", "neutral"]
    issue_type: Optional[str] = Field(
        default=None,
        description="问题类型，如无问题则为 None"
    )
    urgency_score: int = Field(
        ge=1, le=5,
        description="紧急程度，1=最低，5=最高"
    )

# OpenAI SDK 直接支持 Pydantic 模型
response = client.beta.chat.completions.parse(
    model="gpt-4o-2024-08-06",
    messages=[...],
    response_format=OrderAnalysis,
)
result = response.choices[0].message.parsed
```

**TypeScript/Node.js：Zod**

```typescript
import { z } from "zod";

const OrderSchema = z.object({
  orderId: z.string().describe("订单号，格式为 ORD-XXXXXXXX"),
  sentiment: z.enum(["positive", "negative", "neutral"]),
  issueType: z.string().nullable().describe("问题类型，如无问题则为 null"),
  urgencyScore: z.number().min(1).max(5),
});

type Order = z.infer<typeof OrderSchema>;
```

**关键工程实践**：验证应该在模型输出后立即发生，验证失败应触发重试逻辑（建议最多 2 次重试），超过重试次数后进入人工处理队列，而不是静默失败。

### 幻觉在结构化输出里的特殊表现

结构化输出中的幻觉与自然语言不同，更具有隐蔽性：

**格式正确但值错误**：JSON 结构完全符合 schema，但字段值是模型编造的。例如，要求提取"用户提到的产品型号"，当用户根本没提到型号时，模型可能填入一个"看起来合理"的型号，而不是 null。

**预防措施**：
1. 将可能不存在的字段明确标记为 nullable，并在 description 中说明"若无相关信息，返回 null，不要猜测"
2. 对高风险字段（金额、ID、时间）设计额外的验证规则（格式正则、范围约束）
3. 在生产系统中，对关键字段建立置信度抽检机制

## 能力边界与局限

1. **强制 JSON mode 不等于零格式错误**：即使使用 OpenAI 的 Structured Outputs 功能，对于极复杂的嵌套 schema（10+ 字段、3+ 嵌套层），仍有约 2-5% 的格式异常率。
2. **schema 复杂度与延迟正相关**：schema 字段越多，模型推理时间越长，P99 延迟可能增加 30-50%。
3. **不同模型表现差异大**：GPT-4o 的 schema 遵循率约 98%+，而一些较弱的开源模型可能只有 70-80%。
4. **无法消除语义幻觉**：结构化输出解决的是格式问题，不能解决内容幻觉问题。

## PM 视角关键结论

1. **schema 的 description 字段写得越详细，模型输出质量越高**。这不是文档，是"给模型看的说明书"，每个枚举值、每个可选字段都应该有清晰的说明，包括"什么情况下选这个值"。

2. **枚举值比自由文本更可靠**。能用枚举值的地方就不要用自由文本。把 10 个可能的情感类别设计成 enum，比让模型自由生成分类标签，系统稳定性提升一个数量级。

3. **必须有 fallback 机制**。100% 的结构化输出准确率在生产环境中不现实。应该在产品设计阶段就考虑：解析失败时怎么办？是触发重试、走人工处理，还是返回默认值？这个 fallback 逻辑和 AI 调用逻辑同等重要。

4. **schema 版本管理要提前建立**。随着产品迭代，schema 会不断演化。如果没有版本管理，历史数据和新数据的结构可能不兼容，造成数据分析困难。

5. **结构化输出是 AI 系统可观测性的基础**。统一的输出格式，才能方便地统计准确率、错误分布、字段填充率等指标，进而指导 prompt 和 schema 的持续优化。

## 常见误区

**误区一：有了 JSON Mode 就不需要 prompt 中的格式说明**

JSON Mode 确保输出是合法的 JSON，但不能确保字段语义正确。如果 prompt 中没有清晰说明每个字段的含义，模型仍然会按自己的理解填充字段值。JSON Mode 和详细的 prompt 说明是互补的，不是替代关系。

**误区二：schema 越全面越好**

过于全面的 schema（30+ 字段）会导致两个问题：一是模型"格式负担"过重，核心字段的准确率反而下降；二是增加了不必要的 token 消耗。应该遵循"只提取你真正会用的字段"原则，未来需要新字段时再迭代 schema。

**误区三：不测试边界情况**

很多团队只测试"正常输入"下的结构化输出。真正的问题往往出现在边界情况：用户输入为空时、输入语言不匹配时、输入信息不足以填充必填字段时。这些边界情况的测试覆盖，往往比优化正常路径更有价值。

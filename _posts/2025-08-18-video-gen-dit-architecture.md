**一句话摘要：** DiT（Diffusion Transformer）架构已全面取代 UNet 成为视频生成的核心骨架，理解它为什么让视频生成比图像难得多、贵得多，是 PM 做产品决策的前提。

---

## 是什么

如果你两年前还在聊 Stable Diffusion，那时候视频生成模型的骨架几乎清一色是 UNet——一个经典的编码器-解码器结构，在图像生成上非常好用。但从 Sora（2024年初）开始，行业发生了一次悄无声息却影响深远的架构迁移：**UNet 退场，DiT（Diffusion Transformer）登场**。

DiT 的全名是 Diffusion Transformer，顾名思义就是把扩散模型的骨干换成了 Transformer。图像领域的 DiT 论文（Peebles & Xie，2022）证明了 Transformer 在扩散任务上能超过 UNet，随后这个思路被迅速扩展到视频域。

今天你能叫出名字的主流视频生成模型——Sora、Veo 3、Kling 2.x、Runway Gen-4、CogVideoX——底层无一例外都是某种形式的 DiT 变体。

---

## 为什么重要

**场景一：解释为什么视频生成还没办法"实时"**

一个 PM 经常被问到：图像生成都能几秒出图了，视频为什么还要等一两分钟？这不是工程问题，而是架构的计算量决定的。理解 DiT 的 attention 机制，你就能给出有说服力的答案，而不是说"技术比较难"。

**场景二：评估不同供应商的技术路线**

Kling 和 Runway 都说自己支持"长视频"，但它们的实现路径完全不同——一个用 3D Causal Attention，一个用滑窗拼接。这两种技术选择对产品的质量上限和成本结构影响完全不同，不懂架构就没法判断供应商说的是不是真的。

**场景三：预判产品能力边界**

时序一致性差会导致视频闪烁、人脸变化、物体凭空消失。这些问题不是 bug，是当前架构的固有局限。提前理解这个边界，产品设计上就会主动规避，而不是等用户投诉了才救火。

---

## 核心内容

### DiT 替代 UNet 的本质原因

UNet 的核心问题是感受野有限。它依赖卷积层来捕捉空间关系，在处理单张图像时够用，但视频需要同时建模空间关系（单帧内容）和时间关系（帧与帧之间的运动）。卷积对长程依赖天然不擅长。

Transformer 的 attention 机制则可以让每个 token（在视频里就是每个时空 patch）直接关注序列中任意位置的其他 token。这对视频建模来说是质变：第 1 帧的一个像素块可以直接和第 30 帧的同一区域建立关联。

### 3D Full Attention vs 3D Causal Attention

这是当前视频模型里最重要的架构选择之一，直接影响产品的关键属性：

```
视频 token 空间示意（T帧 × H × W 的时空体）：

[Frame 0] → [Frame 1] → [Frame 2] → ... → [Frame N]
   ↓            ↓            ↓                  ↓
各帧被切成若干 patch，每个 patch 是一个 token

Full Attention（双向）：
Frame 0 的 token A ←→ Frame N 的 token B（可以互相看）

Causal Attention（单向）：
Frame 0 的 token A → Frame N 的 token B（后帧只能看前帧）
Frame N 的 token B ✗→ Frame 0 的 token A（前帧看不到后帧）
```

| 维度 | 3D Full Attention | 3D Causal Attention |
|------|------------------|---------------------|
| 质量上限 | 更高（全局一致性好） | 稍低（无法利用未来帧） |
| 计算量 | O(T²HW) 级别，极贵 | 可流式推理，相对可控 |
| 长视频支持 | 很难（显存爆炸） | 天然支持（流式生成） |
| 典型代表 | Sora（推测）、CogVideoX | Kling、部分 Runway 版本 |

Causal Attention 牺牲了一些全局一致性，换来了长视频的可行性——这就是为什么 Kling 能支持 2-3 分钟长视频，而很多竞品卡在 10-20 秒。

### Flow Matching 为什么比 DDPM 强

经典 DDPM（去噪扩散概率模型）的推理过程需要走几百步去噪，每一步都要过一次大模型，非常慢。

Flow Matching（流匹配）的思路不同：它学习的是从噪声到数据的一条"直线路径"（最优传输意义下），而不是弯弯绕绕的扩散路径。

```
DDPM 路径（示意）：
噪声 → 曲折路径（需要 1000 步）→ 视频

Flow Matching 路径（示意）：
噪声 ─────────────────────────────→ 视频
      （更直的路径，50-100 步够用）
```

实际效果是：相同质量下推理步数减少 5-10 倍。Stable Diffusion 3、Flux、Sora 都采用了 Flow Matching 或其变体（Rectified Flow）。对 PM 来说，这意味着生成速度的提升不是靠压缩质量换来的，而是算法本身更高效。

### VAE 时序编码器的作用

视频不能直接用像素做 attention——一个 5 秒 720p 视频，像素数量是天文数字。VAE（变分自编码器）在这里扮演"压缩器"的角色：把视频压缩到潜空间（latent space），让模型在低维空间做扩散，最后再解码回像素。

视频 VAE 比图像 VAE 多了时序维度的压缩。一个典型的配置是 4×8×8——时间维度压缩 4 倍，空间维度各压缩 8 倍。这意味着 60 帧的视频在潜空间里只有 15 帧，计算量大幅减少，但也引入了时序一致性的额外挑战：解码回来的相邻帧不一定完全匹配。

### 时序一致性的三大挑战

**1. 运动模糊与闪烁**：帧与帧之间亮度或纹理的细微变化，在视觉上会被感知为"闪烁"。这通常来自 VAE 解码的随机性，以及 attention 在时序方向建模不完整。

**2. 物理不真实**：液体流动、布料飘动、刚体碰撞——这些都需要模型理解物理规律。当前模型是从视频数据"学"出来的统计规律，不是真正的物理仿真，所以长序列下物理行为容易失真。

**3. 主体漂移（Identity Drift）**：视频里一个人物的脸，生成到后半段可能悄悄变样了。这是因为 Causal Attention 下，后帧依赖前帧的信息量是衰减的。

### 推理成本拆解

生成一段 10 秒、720p、24fps 的视频，需要约 240 帧。以典型配置估算：

```
计算量来源拆解（粗略）：
- DiT 主干推理（50 步去噪 × 全部 token 的 attention）：~60-70% 成本
- VAE 编码/解码：~10-15% 成本  
- 文本编码器（T5/CLIP）：~5% 成本
- 其他（调度器、后处理）：~10-20% 成本

典型推理时间（A100 单卡）：
- 5s 视频（480p）：~1-2 分钟
- 10s 视频（720p）：~3-5 分钟
- 对比：生成一张 1024px 图像 ~5-15 秒
```

这个成本差异不是硬件不够，是模型规模和序列长度共同决定的数学上限。

## 能力边界与局限

- **长视频质量衰减是已知问题**：超过 30 秒后，几乎所有模型的一致性都会下降，目前没有完美解法
- **文本到视频的语义对齐仍然有限**：复杂的多步骤动作描述（"先走两步，然后转身，再坐下"）往往只能生成其中一个动作
- **4K 分辨率几乎不可行**：计算量随分辨率平方增长，当前架构做 4K 视频生成代价极高
- **训练数据版权问题**：这是行业整体面临的法律灰色地带，不是技术问题，但影响商业可行性

---

## PM 视角关键结论

1. **视频比图像贵 10-50 倍不是借口，是物理事实**——序列长度带来的 attention 计算量决定了这个成本结构，短期内不会根本改变，产品定价必须反映这个现实。

2. **选择供应商时，问清楚 Causal 还是 Full Attention**——前者更适合需要长视频的场景（广告、教育），后者在短视频质量上有优势，但成本更高。

3. **Flow Matching 是判断模型代际的标志之一**——还在用 DDPM 的模型，速度上已经输了一个档次，除非有其他特殊优势。

4. **时序一致性问题应该写进产品使用限制里**——而不是让用户靠碰运气发现，否则退款投诉是必然的。

5. **当前最值得期待的技术突破点是视频 VAE 的改进**——更好的时序编码器能在不增加计算量的前提下显著改善闪烁和漂移问题，这个方向正在快速进展。

## 常见误区

**误区一：参数量越大，视频越好**

不对。视频质量更多取决于训练数据质量、时序建模设计和后训练对齐（RLHF/DPO），不是纯粹的参数规模。一个 5B 参数的视频模型完全可能在运动流畅度上超过 20B 的竞品。

**误区二：生成速度慢只是推理优化没做好**

速度问题的本质是 attention 计算的二次方复杂度，不是工程优化不够。当然量化、蒸馏、推理缓存可以改善，但有数学下限，不可能通过工程手段无限提速。

**误区三：买到 GPU 就能自己部署这些模型**

主流视频生成模型（Sora、Veo 3）的权重根本不对外开放。开源替代（CogVideoX、Open-Sora）在质量上差距明显。即使是 Kling 这类有 API 的产品，也无法私有化部署。PM 在供应商策略上要对这一点有清醒认识。

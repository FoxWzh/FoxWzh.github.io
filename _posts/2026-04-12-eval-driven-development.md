**在AI产品开发中，没有评估集就开始迭代，相当于在黑暗中开车——你不知道自己在前进还是在倒退。**

## 是什么

Eval-Driven Development（EDD，评估驱动开发）是一种AI产品迭代方法论：在优化模型或修改Prompt之前，先定义并构建评估集（eval set），用评估结果来客观衡量每一次变更的效果。

这个思路来源于软件工程中经典的测试驱动开发（TDD），但针对AI系统的特殊性做了适配。在传统TDD中，你先写一个会失败的测试，再写代码让它通过。在EDD中，你先定义"什么叫做好的AI输出"，构建能够测量这个标准的评估集，再去修改Prompt或微调模型，以评估分数的提升来验证改进效果。

**EDD和TDD的核心共同点：** 在做任何改动之前，先有一个客观的、可重现的衡量标准。这个标准先于实现而存在，而不是事后凑数。

**EDD和TDD的关键不同：** TDD的测试是确定性的（代码要么通过要么失败），EDD的评估是概率性的（AI输出的好坏往往是连续分布，不是二元判断）。这使得EDD在工程实施上更复杂，但也正因如此更需要系统化方法。

## 为什么重要：三个产品场景

**场景一：Prompt优化反复横跳**

某AI写作工具团队在优化"摘要生成"功能。PM根据几个用户投诉修改了Prompt，发布后收到更多投诉。于是再次修改，情况又变了……三个月内修改了23个版本，但团队无法回答"现在的版本比三个月前好还是差？"

这就是没有eval的典型症状：每次改动都是基于直觉，无法客观衡量，更无法防止退化（regression）。

**场景二：模型切换引发未知影响**

某法律AI助手从GPT-4 Turbo迁移到GPT-4o，原因是成本降低了40%。切换后，工程师测试了10个样例，"感觉差不多"，于是上线。

两周后，客户开始反映某类合同条款的解析准确率明显下降。经过调查，发现GPT-4o在这个细分场景下的表现确实弱于GPT-4 Turbo——但这10个测试样例恰好没有覆盖到这个edge case。

如果有覆盖足够广的eval set（包含150个真实法律场景的golden set），这个退化问题会在上线前被检测到。

**场景三：功能迭代速度受限**

某客服AI产品，每次想更新Prompt，都需要人工测试200个case，耗时3天，还需要2名专业客服人员参与评审。这导致产品迭代周期从预期的1周被迫拉长到3-4周，严重制约了产品竞争力。

引入自动化eval后，80%的回归测试可以在30分钟内自动完成，人工评审仅需聚焦在20%的高风险场景，迭代周期压缩到了原来的三分之一。

## 核心内容

### Eval的三种类型

**类型一：单元Eval（Unit Eval）**

针对单条输入输出对，用确定性规则进行判断。类比于单元测试。

适用场景：
- 格式验证（输出是否符合JSON schema？）
- 长度限制（摘要是否在100字以内？）
- 关键词存在性（回答中是否包含必须声明的免责声明？）
- 精确匹配（工具调用参数是否正确？）

优点：运行速度极快（毫秒级），成本几乎为零，结果完全可重现。
局限：只能评估有明确正确答案的维度，无法捕获语义质量。

**类型二：集成Eval（Integration Eval）**

模拟真实用户场景的端到端评估，测试完整的交互链路。类比于集成测试。

适用场景：
- 多轮对话的连贯性测试
- RAG系统的检索+生成全链路测试
- Agent任务的端到端完成率测试
- 跨功能模块的协同测试

优点：最接近真实用户体验，能发现跨组件的问题。
局限：运行慢、成本高（每次可能消耗数百到数千token），难以定位具体失败原因。

**类型三：对比Eval（Comparative Eval）**

将两个版本（A版本 vs B版本）的输出放在一起，用LLM或人工判断哪个更好。类比于A/B测试。

适用场景：
- Prompt版本对比（v1 vs v2哪个更好？）
- 模型切换评估（GPT-4o vs Claude 3.5 Sonnet哪个更适合？）
- 细粒度质量维度对比（哪个回答更流畅？哪个更准确？）

优点：不需要绝对标准答案，适合评估"整体质量"这类主观维度。
局限：存在位置偏差和比较对象污染问题，需要随机化处理。

### Golden Set的构建原则

Golden Set（黄金测试集）是eval的核心资产，也是AI产品最重要的知识沉淀之一。构建原则：

**原则一：覆盖Happy Path（正常路径）**

60-70%的case应该覆盖最常见的用户意图和典型输入。这是确保核心功能稳定的基础。

例如，对于一个合同摘要功能，happy path包括：标准条款的准确摘要、关键日期的提取、甲乙双方责任的识别。

**原则二：覆盖Edge Case（边界情况）**

20-25%的case应该覆盖输入的极端情况：非常短的输入、非常长的输入、包含特殊字符的输入、用户意图模糊的输入、多语言混合输入。

Edge case往往是模型行为最不稳定、用户体验最差的地方，也是最容易在版本迭代中发生退化的地方。

**原则三：覆盖Regression Case（回归情况）**

5-15%的case来自历史上出现过的真实bug。每次修复一个线上问题，就把这个case加入golden set。

这个部分随着产品迭代会持续增长，是防止历史bug复现的关键防线。

**原则四：定期刷新**

Golden set不是一次性构建，而是需要定期更新：每季度审查一次，移除过时的case，补充新出现的用户场景。一个2年没有更新的golden set，很可能已经无法反映真实用户行为。

### 常见Eval指标详解

**ROUGE（Recall-Oriented Understudy for Gisting Evaluation）**：通过词汇重叠度来衡量生成文本与参考文本的相似性。ROUGE-1衡量单词级别重叠，ROUGE-L衡量最长公共子序列。

优点：完全自动化，无成本。缺点：无法理解语义，一个意思完全相同但用词不同的回答会得到低分。适合：摘要任务、翻译任务的初步筛选。

**LLM-as-Judge**：用一个强大的LLM（如GPT-4o、Claude 3.7 Sonnet）来评估另一个模型的输出质量。这是当前业界最广泛采用的高质量自动化eval方法。可以评估流畅性、准确性、有用性等主观维度，与人类评分相关性达0.7-0.85。

**人工评分（Human Evaluation）**：由领域专家或产品团队成员对模型输出进行打分。最准确，但成本高、速度慢，难以大规模应用。通常作为LLM-as-Judge的校准工具，或用于最终上线前的质量确认。

**任务完成率（Task Completion Rate）**：在有明确目标的Agent或多轮对话场景中，模型成功完成用户任务的比例。这是最接近业务价值的eval指标，但定义"完成"的标准本身就需要仔细设计。

### Eval自动化 vs 人工Eval的权衡

```
成本与可靠性权衡矩阵

┌─────────────────┬──────────────────┬──────────────────┬──────────────────┐

├─────────────────┼──────────────────┼──────────────────┼──────────────────┤
│  规则评估       │  极快（<1s）     │  极低（接近零）  │  高（客观一致）  │

├─────────────────┼──────────────────┼──────────────────┼──────────────────┤

```

### CI/CD中集成Eval的流程设计

将eval集成到CI/CD管道中，是实现"eval即门控"的关键一步。基本流程如下：

```
EDD 开发迭代循环

┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  1. 定义问题                                                    │
│     └─ 明确要改进的具体指标（如：摘要准确率从75%提升到85%）    │
│                                                                 │
│  2. 构建/扩充 Golden Set                                        │
│     └─ 覆盖 happy path + edge case + regression case           │
│                                                                 │
│  3. 运行基线评估（Baseline Eval）                               │
│     └─ 记录当前版本在 golden set 上的得分                      │
│                                                                 │
│  4. 实施变更（修改 Prompt / 调整参数 / 换模型）                 │
│     └─ 基于假设进行改动，记录改动意图                          │
│                                                                 │
│  5. 运行对比评估                                                │
│     └─ 新版本 vs 基线，计算指标变化                            │
│                                                                 │
│  6. 决策                                                        │
│     ├─ 分数提升 + 无退化 → 合并到主分支，推进上线             │
│     ├─ 部分指标提升但有退化 → 权衡取舍，可能需要再迭代        │
│     └─ 分数无改善或下降 → 丢弃变更，重新分析假设              │
│                                                                 │
│  7. 集成到 CI/CD                                                │
│     └─ 每次 PR 自动运行 eval，低于阈值自动阻断合并            │
│                                                                 │
│  ↑─────────────────────────────────────────────────────────────┘
│                     循环迭代                                    │
└─────────────────────────────────────────────────────────────────┘
```

**CI/CD集成的实践要点：**
- 快速eval（规则+轻量LLM评估）作为每次PR的强制门控，运行时间控制在5分钟内
- 完整eval（含LLM-as-Judge的全量golden set评估）作为每日定时任务或每次合并到main分支时运行
- 设置明确的阈值：核心指标下降超过2%为blocking，下降1-2%为warning需要人工确认
- Eval结果自动写入PR评论，方便reviewer查看

## 能力边界与局限

**Golden set存在时间衰减**：随着产品发展、用户群体变化，golden set中的case可能逐渐偏离真实用户行为。一个2年前构建的评估集，很可能无法反映今天的用户期望。需要建立持续更新机制。

**Eval指标可以被过度优化**：如果把eval分数作为唯一优化目标，存在"Goodhart's Law"风险——当一个指标成为目标，它就不再是好指标。过度拟合eval set会导致在真实用户场景中表现下降，但在eval上得分很高。

**构建高质量golden set本身成本很高**：对于复杂的专业领域（如医疗、法律），构建准确的golden set需要领域专家参与，成本可能高达数万到数十万元。这也是为什么golden set是核心资产。

**Eval无法完全替代用户研究**：定量eval很难捕获用户体验的定性维度，如回答是否让用户感到被理解、对话风格是否符合品牌调性。用户访谈和可用性测试仍然是必要的补充。

## PM视角关键结论

1. **Golden set是AI产品最重要的技术资产之一，与代码同等重要。** 一套高质量的200-500条golden set，比任何单次的Prompt优化都更有价值。它是团队认知的结晶，是防止退化的护盾，是加速迭代的基础设施。

2. **没有eval就没有可靠迭代，有感觉没数据的迭代最终都会走弯路。** 对AI产品来说，"感觉更好了"是最危险的四个字。你的感觉可能是正确的，也可能是因为你只看了几个你期望看到的例子。只有eval数据才能告诉你真相。

3. **Eval覆盖率决定产品迭代速度的上限。** 自动化eval覆盖率越高，每次改动的验证成本越低，迭代频率就越高。那些迭代速度快的AI产品团队，背后一定有完善的eval基础设施。

4. **Eval应该先于功能开发存在。** 在开始优化一个AI功能之前，先问：我们用什么标准判断这次优化是否成功？如果这个问题答不上来，就不应该开始优化。

5. **Eval结果要与业务指标双向验证。** Eval分数提升应该能在一定程度上预测业务指标（用户满意度、任务完成率）的提升。如果eval分数提升了，但业务指标没有变化，说明eval指标设计有问题，需要重新审视。

## 常见误区

**误区一："我们的场景太复杂，没办法评估"**

没有不能评估的AI场景，只有还没有想清楚如何评估。即使是最复杂的创意写作或开放式对话，也可以从某个维度切入（如相关性、安全性、格式符合度）开始构建eval。从最简单、最确定的维度开始，逐步扩展。

**误区二："有了eval，就不需要人工测试了"**

Eval是加速和规模化人工判断的工具，不是替代品。LLM-as-Judge的0.7-0.85与人类相关性意味着仍有15-30%的case需要人工确认。上线前的人工评审永远不应该被完全省掉，尤其是高风险场景。

**误区三："eval分数是越高越好的绝对指标"**

Eval分数是相对指标，意义在于版本之间的对比，而不是绝对值。一个ROUGE-1分数0.6的摘要系统可能已经是行业顶尖水平，也可能还远未达到可用标准——这取决于你的基线和任务特性。永远要结合业务上下文来解读eval数字。

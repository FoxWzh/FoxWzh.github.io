**Prompt Engineering 不是一次性的配置工作，而是 AI 产品迭代的持续竞争力——掌握从零样本到思维链的核心技法，是 PM 构建可靠 AI 产品的基础能力。**

## 是什么

Prompt Engineering（提示工程，简称 PE）是指通过设计和优化输入给大语言模型的文本指令，来引导模型产出符合预期的输出。它不涉及模型权重的修改，而是在推理阶段通过文本设计来控制模型行为。

从产品视角来看，PE 是介于"用户需求"和"模型能力"之间的翻译层。一个功能强大的底层模型，如果 prompt 设计粗糙，产品体验可能一塌糊涂；反之，合理的 PE 设计可以让中等能力的模型表现出远超预期的效果。

2024 年至 2025 年，随着 GPT-4o、Claude 3.5 Sonnet、Gemini 1.5 Pro 等模型的广泛商用，PE 从研究领域正式进入产品工程主流。几乎所有 AI 产品背后都有一套精心设计的 prompt 体系，这套体系的质量直接决定了产品的可靠性和用户体验。

## 为什么对 PM 重要

**场景一：客服机器人的回答质量控制**

某金融科技公司的智能客服项目，最初上线时用户满意度只有 52%。问题根源不在模型选型，而在 prompt 设计。系统提示只有一句"你是客服助手，帮助用户解决问题"。经过三轮 PE 迭代，加入角色定义、负向约束、few-shot 示例后，满意度提升至 78%。这 26 个百分点的差异，全部来自 prompt 的改进，没有更换任何模型。

**场景二：代码生成工具的一致性问题**

某开发工具团队发现，他们的代码生成功能对于同一个需求，每次输出的代码风格都不一样——有时用 async/await，有时用 Promise，有时混用。通过引入结构化提示和 few-shot 示例，将团队的代码风格规范编码进 prompt，代码一致性从 40% 提升至 85%。

**场景三：内容生成的品牌调性**

一家媒体公司使用 AI 辅助撰写新闻摘要，初期生成的内容语气偏口语化，与品牌调性不符。通过 Persona 设计和 Role Prompting，明确定义"资深财经记者"的写作风格，并提供 5 个品牌调性典型样本作为 few-shot 示例，品牌调性符合率从 35% 提升至 81%。

## 核心技法详解

### 1. Zero-shot：直接指令

零样本提示是最基础的形式——直接告诉模型你要它做什么，不提供任何示例。

```
请将以下中文文本翻译成英文：
"人工智能正在改变我们的工作方式。"
```

**适用场景**：任务描述清晰、模型训练数据中大量覆盖的任务（翻译、简单分类、摘要）。对于模糊任务或模型未充分训练的领域，Zero-shot 容易产生漂移。

**产品实践**：Zero-shot 适合作为 MVP 快速验证，但不适合作为生产环境的最终方案。

### 2. Few-shot：示例驱动学习

Few-shot 是在 prompt 中提供 2-8 个输入输出示例，让模型学习期望的格式和风格。这是 PE 中性价比最高的技法。

```
将用户评价分类为：正面/负面/中性

示例1：
用户评价："物流很快，包装完整，非常满意"
分类：正面

示例2：
用户评价："质量一般，价格还行"
分类：中性

示例3：
用户评价："收到的是破损商品，完全无法使用"
分类：负面

现在请分类：
用户评价："{{user_review}}"
分类：
```

**关键洞察**：Few-shot 的例子质量比数量更重要。3 个高质量、覆盖边界情况的示例，比 10 个泛泛的示例效果更好。在实际产品中，负责维护 few-shot 示例库的应该是 PM 和业务团队，而不仅仅是工程师。

### 3. Chain-of-Thought（CoT）：思维链

CoT 是要求模型在给出最终答案之前，先写出推理步骤。由 Google Brain 2022 年的研究正式提出，对复杂推理任务有显著提升。

```
某电商平台，上月 GMV 为 1200 万元，本月增长了 15%，
但退款率从 3% 上升到 5%。
请计算本月实际净收入变化了多少？
请一步步计算。
```

模型会先计算本月 GMV（1380 万），再计算上月净收入（1164 万），再计算本月净收入（1311 万），最终得出结论。如果不用 CoT，模型可能直接给出一个计算错误的答案。

**产品影响**：CoT 会显著增加 token 消耗（通常增加 30%-80% 的输出 token），从而增加 API 成本和延迟。PM 需要权衡：对于决策型、分析型任务值得用 CoT；对于简单检索型任务则是浪费。

### 4. Zero-shot CoT："让我们一步步思考"

只需在 prompt 末尾加上"Let's think step by step"或"请一步步思考"，就能激活模型的链式推理能力，无需提供任何示例。

这是 Kojima et al. 2022 年发现的"神奇咒语"效应，在 GPT-3 时代效果尤为显著。在 GPT-4 和 Claude 3 等更强的模型上，效果有所降低，但在处理计算题、逻辑题时仍然有效。

### 5. Role Prompting 与 Persona 设计

通过赋予模型特定的角色身份，影响其输出风格和知识侧重。

**浅层 Persona**（不推荐）：
```
你是一个有帮助的助手。
```

**深层 Persona 设计**（推荐）：
```
你是一位拥有 15 年经验的 B2B SaaS 产品顾问。
你服务过 Salesforce、HubSpot 等头部企业的产品团队。
你的沟通风格是直接、数据导向的，会主动指出方案的潜在风险。
你用第一人称回答，不使用"当然"、"绝对"等口头禅。
```

深层 Persona 设计需要明确：背景经历、沟通风格、知识领域边界、禁用表达。

### 6. 结构化提示：XML 和 JSON 组织指令

当 prompt 包含多个组成部分（角色、任务、约束、示例、输入）时，使用结构化标签可以显著提升模型的理解准确率。

```xml
<system>
  <role>你是一位专注于 SaaS 产品的增长分析师</role>
  <constraints>
    - 只分析定量数据，不做主观猜测
    - 数字保留两位小数
    - 回答长度控制在 200 字以内
  </constraints>
</system>

<task>分析以下用户留存数据，找出异常点</task>

<data>
{{retention_data}}
</data>
```

这种结构化方式在 Claude 系列模型上效果尤为显著，因为 Anthropic 在训练时就使用了大量 XML 结构的数据。

### 7. 负向指令的有效性与局限

"不要做 X"类的指令存在一个根本性问题：大语言模型在处理否定时并不像规则引擎那样可靠。

研究显示，在复杂 prompt 中，负向指令的遵循率平均比正向指令低 20-35%。

**不可靠的写法**：
```
不要使用专业术语
不要超过 100 字
不要给出具体建议
```

**更可靠的正向替代**：
```
使用初中生能理解的日常用语
回答控制在 80-100 字之间
只列举事实，不做建议
```

### 8. Temperature 参数的产品含义

Temperature 控制模型输出的随机性（0 = 确定性最高，1 = 随机性最高，部分模型支持到 2）。

| 任务类型 | 推荐 Temperature | 原因 |
|---------|----------------|------|
| 数据提取、分类 | 0 - 0.2 | 需要稳定、可重复的输出 |
| 问答、总结 | 0.3 - 0.5 | 平衡准确性与表达多样性 |
| 内容撰写、营销文案 | 0.6 - 0.8 | 需要创意和语言变化 |
| 头脑风暴、创意生成 | 0.8 - 1.0 | 最大化多样性 |

**产品陷阱**：很多团队把 Temperature 设置为 0.7 当作"默认值"，这对于分类任务来说是一个严重错误，会导致相同输入得到不同分类结果，让产品行为不可预期。

## 各技法适用场景矩阵

```
                    输出可控性要求
                    低 ←————————————→ 高
                    |                  |
高 ┌───────────────┼──────────────────┐
   │               │                  │
任 │  CoT + 创意写作│  CoT + 结构化输出 │
务 │  (高复杂、低控) │  (高复杂、高控)   │
复 │               │                  │
杂 ├───────────────┼──────────────────┤
度 │               │                  │
   │  Zero-shot    │  Few-shot        │
   │  创意头脑风暴  │  分类/提取/格式化  │
低 │  (低复杂、低控) │  (低复杂、高控)   │
   └───────────────┴──────────────────┘

技法选择参考：
  Zero-shot     → 快速验证、通用任务
  Few-shot      → 格式控制、风格统一、分类任务
  CoT           → 多步推理、复杂计算、决策分析
  Zero-shot CoT → 无示例可用时的推理增强
  Role/Persona  → 风格控制、专业知识侧重
  结构化提示    → 多指令组合、复杂系统 prompt
```

## 能力边界与局限

**PE 无法解决的问题**：

1. **知识截止**：模型不知道的事实，再好的 prompt 也无法凭空生成正确答案。
2. **算术可靠性**：CoT 能提升但不能消除数学计算错误，复杂计算仍需外挂计算器工具。
3. **长文本一致性**：即使用了 CoT，超过 2000 字的生成内容仍然容易在后半段出现逻辑不一致。
4. **对抗性注入**：精心设计的 prompt injection 攻击，现有 PE 技术无法完全防御。
5. **跨会话记忆**：PE 只能控制单次推理，无法替代持久化记忆系统。

**边界认知**：PE 是在模型能力范围内的优化，不是能力扩展。当 PE 已经优化到极限但效果仍不满足要求时，需要考虑微调、RAG 或更换底层模型。

## PM 视角关键结论

1. **PE 是产品迭代的持续工作，不是一次性配置**。上线只是起点，应该建立 prompt 版本管理机制，像管理代码一样管理 prompt，记录每次变更的原因和效果数据。

2. **Few-shot 示例是性价比最高的质量杠杆**。维护一个高质量的示例库（覆盖正常情况、边界情况、反例），往往比升级到更贵的模型效果更好，且成本更低。

3. **CoT 对推理密集型任务效果显著，但要计算 token 成本**。在 GPT-4o 上，CoT 可能将单次请求成本增加 50%。对于高频调用场景，需要评估 ROI。

4. **负向指令应转化为正向约束**。将"不要"改为"只做"，将"避免"改为"确保"，可以将指令遵循率平均提升 20%。

5. **Temperature 设置是产品决策，不是技术参数**。分类任务用 0.1，创意任务用 0.7，这个区别对产品可靠性有直接影响，PM 应该参与这个决策，而不是完全交给工程师。

## 常见误区

**误区一：prompt 越长越好**

很多团队认为把所有可能的情况都写进 prompt 就能提高质量。实际上，超过 1500 字的 prompt 会导致注意力分散（attention drift），模型反而会忽略中间部分的指令。关键指令应该简洁并放在首部或尾部。

**误区二：找到一个好 prompt 就一劳永逸**

一个在 GPT-4 上表现良好的 prompt，切换到 Claude 3.5 后可能需要重新调整。模型升级（如从 GPT-4 到 GPT-4o）也可能导致原有 prompt 效果退化。PE 工作需要伴随模型迭代持续维护。

**误区三：PE 可以替代产品设计**

有些团队试图用复杂的 prompt 来"修补"产品设计缺陷，比如用 prompt 来处理模糊的用户意图。这会导致 prompt 越来越臃肿，维护成本指数增长。PE 应该建立在清晰的产品定义之上，而不是弥补产品设计的缺失。

> 一句话摘要：AI"胡说八道"不是一种病，是三种病——搞清楚是哪种，才能用对药。

---

## 是什么

**幻觉（Hallucination）** 是指大模型生成了听起来合理、但实际上错误或无法核实的内容。

很多人以为幻觉就是"模型在瞎编"，但 2024 年学术界和工业界已经把它拆成了三种截然不同的类型：

- **事实性幻觉**：知识是错的
- **推理性幻觉**：逻辑是错的
- **格式性幻觉**：结构是错的

这三种幻觉的成因不同，影响场景不同，解决方案也完全不同。

---

## 为什么重要

### 场景 1：法律助手给出了一条不存在的法规

用户问"根据《劳动法》第XX条……"，模型一本正经引用了一条实际上不存在的条款。这是**事实性幻觉**，用 RAG 接入真实法律数据库可以解决。

### 场景 2：财务分析工具算错了账

模型引用的数字都是正确的，但中间的计算步骤出了错，最终结论不对。这是**推理性幻觉**，换个模型或用 CoT 提示可以改善。

### 场景 3：Agent 工具调用时传了错误的参数格式

让模型输出 JSON 调用某个 API，它输出的 JSON 里少了一个必填字段，或者类型写错了。这是**格式性幻觉**，用结构化输出约束可以解决。

三种幻觉，三种解法。不分清楚就会用错药。

---

## 核心内容

### 第一类：事实性幻觉

**定义**：模型生成的内容与客观现实、已知事实或给定资料不符。

细分为三种：

**① 凭空捏造（Fact Fabrication）**

模型"发明"了一个不存在的东西：虚构的论文、不存在的公司、没说过的名人名言。

> 典型案例：律师用 ChatGPT 写诉状，模型引用了多篇根本不存在的判例，最终被法官处罚。（2023 年真实事件）

**② 事实错位（Fact Inconsistency）**

真实的要素都存在，但被张冠李戴：把 A 公司的数据安在 B 公司头上，或把某件事的时间、地点搞混。

**③ 知识陈旧（Outdated Knowledge）**

模型的训练数据有截止日期，对截止日期之后发生的事情一无所知，但它不会说"我不知道"，而是会用旧信息填充——而且语气很自信。

**根本原因**：模型是概率机器，它学的是"什么词在什么语境下出现的概率高"，而不是"这句话是不是真的"。训练数据里如果某个错误说法反复出现，模型就会把错误当正确学进去。

---

### 第二类：推理性幻觉

**定义**：模型掌握的事实是对的，但推理过程出了错，导致结论错误。

**① 逻辑断裂（Logical Inconsistency）**

前后矛盾，或推导步骤中出现不合逻辑的跳跃。

**② 计算错误（Mathematical Error）**

数字是对的，公式用错了。比如要求计算复利，模型用了单利公式。这在表面上看不出来，但结果是错的。

**③ 多步推导雪崩（Step Drift）**

在链式推理中，第一步出了微小偏差，后续每一步都基于错误前提继续推导，最终结论离正确答案越来越远——就像导弹飞出去角度偏了一度，落点差了十公里。

**④ 逆向推理失败（Reversal Curse）**

2024 年被广泛讨论的一个现象：模型知道"A 是 B 的孩子"，但问"B 的孩子是谁"时却答不上来。模型学的是文本中的顺序关系，而不是真正理解了因果和反向推理。

**根本原因**：模型的推理能力本质上仍是基于模式匹配，而不是真正的逻辑运算。它看起来在"思考"，其实是在"续写"。

---

### 第三类：格式性幻觉

**定义**：内容可能是对的，但输出的结构或格式不符合要求。

这类幻觉在 2024 年随着 AI Agent 和工具调用的普及变得越来越重要。

**① 结构失效（Structural Invalid）**

要求输出 JSON，它输出了带注释的 JSON（不合法）、多余的 markdown 包裹、或者括号没闭合。这种错误会直接导致下游代码解析崩溃。

**② 工具调用参数错误（Tool Call Error）**

在 Agent 场景里，模型调用外部工具时传了错误的参数名、缺少必填字段、或数据类型不对。

**③ Schema 漂移（Schema Drift）**

模型输出的 JSON 结构看起来对，但字段名和约定的不一样（比如约定是 `user_id`，它输出了 `userId`），或者多出了没有定义的字段。

**根本原因**：模型对"格式"的理解是统计性的，而不是规则性的。它看过大量 JSON，但它理解 JSON 的方式是"这种结构常见"，而不是"这是一套必须严格遵守的语法规则"。

---

### 三类幻觉对比

| | 事实性幻觉 | 推理性幻觉 | 格式性幻觉 |
|--|-----------|-----------|-----------|
| **核心问题** | 知识是错的 | 逻辑是错的 | 结构是错的 |
| **典型表现** | 引用不存在的内容 | 计算/推导结论错误 | JSON 格式不合法 |
| **高风险场景** | 医疗、法律、新闻 | 财务、数学、代码 | Agent、API 调用 |
| **主要解法** | RAG、知识库接地 | CoT、推理模型、验证 | 结构化输出、Schema 约束 |
| **用户感知** | 难发现，容易被骗 | 中等，需要专业知识才能发现 | 容易发现，直接报错 |

---

### 2024 年主流缓解策略

**对事实性幻觉**：
- **RAG（检索增强生成）**：不让模型"背题"，而是"开卷考试"，用检索到的真实文档作为回答依据
- **引用来源 UI**：让模型在回答旁标注出处，用户可以自行核实
- **知识截止提示**：在 System Prompt 里明确告知模型"你的训练截止于 XX，请对时效性内容保持谨慎"

**对推理性幻觉**：
- **Chain-of-Thought（思维链）**：让模型把推理过程一步步写出来，便于发现错误
- **Self-Consistency（自洽采样）**：同一问题采样多次，取多数一致的答案
- **使用推理模型**：o1、R1 等专门训练了推理能力的模型在这类问题上表现更好

**对格式性幻觉**：
- **结构化输出（Structured Output）**：OpenAI、Anthropic 等平台提供 JSON Schema 约束模式，强制输出符合格式
- **后处理验证**：用代码层对输出进行 schema 验证，不合法就重试

## 能力边界 / 局限

- **三类幻觉没有完美解法**。RAG 能减少事实性幻觉，但检索本身也可能召回错误内容；CoT 能暴露推理错误，但不能保证推理正确。
- **幻觉率与任务难度正相关**。简单的问题几乎不会幻觉；复杂、边缘、跨领域的问题幻觉率会显著升高。
- **模型越自信，幻觉越危险**。幻觉内容通常语气笃定、措辞流畅，比真实错误更难被发现。
- **幻觉检测本身很难**。目前没有完全可靠的自动幻觉检测方案，LLM-as-Judge 方法本身也会产生幻觉。

---

## PM 视角的关键结论

1. **上线前必须明确：你的产品对哪类幻觉容忍度最低**。医疗类产品事实性幻觉是红线；财务类产品推理性幻觉是红线；Agent 类产品格式性幻觉是红线。

2. **RAG 不是万能药**。它能解决事实性幻觉，但对推理性和格式性幻觉没有直接帮助。用之前先搞清楚你要解决的是哪类问题。

3. **格式性幻觉是 Agent 产品最容易忽视的坑**。一旦输出格式不对，下游逻辑直接崩溃，用户体验极差，但在 demo 阶段往往暴露不出来。

4. **用评测集量化幻觉率，不要靠主观感受**。为你的核心场景建立专属测试集，每次模型升级都跑一遍，幻觉率是可以被量化追踪的指标。

5. **告知用户模型的局限性**，既是合规要求，也能降低幻觉带来的负面体验。"AI 可能出错，请核实重要信息"不是推卸责任，是负责任的产品设计。

## 常见误区

**误区 1：用 RAG 就能解决所有幻觉**

RAG 主要针对事实性幻觉，对推理性和格式性幻觉基本无效。而且 RAG 的检索质量本身也是问题——召回了错误的文档，反而会加重幻觉。

**误区 2：模型越大，幻觉越少**

参数量大的模型在知识覆盖上确实更好，但推理性幻觉并不随模型规模线性下降。o1 这类专门训练推理的模型比单纯更大的模型在推理任务上表现更好。

**误区 3：幻觉是 bug，会被修复**

幻觉是当前 Transformer 架构的系统性特征，不是可以"打补丁"修复的 bug。它会随着训练改善，但不会消失——这是目前所有主流模型的共同局限。

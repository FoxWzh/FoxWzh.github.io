**一句话摘要：** HumanEval已经基本没有区分意义，SWE-bench才是真正测试AI能否替代程序员的关键评测，理解50%和60%之间的实际差距，直接影响AI编程助手的产品决策。

---

## 是什么

代码评测是最容易被误读的评测类别之一——因为"代码能力"本身差异极大：写一个排序函数、补全一行代码、修复一个真实Bug、理解十万行代码库里的架构逻辑，这些都叫"代码能力"，但难度相差了几个数量级。

评测体系的演化，恰好沿着这个难度轴在推进：从函数级别的HumanEval，到文件级别的MBPP，再到仓库级别的SWE-bench，每一步都更接近程序员的真实工作场景。

---

## 为什么重要

**场景一：选择AI编程助手工具**

GitHub Copilot、Cursor、Windsurf、CodeWhisperer——市场上的AI编程助手越来越多。评测分数是筛选工具的客观参考，但你需要知道哪些分数有意义，哪些只是营销数字。

**场景二：规划AI辅助开发的ROI**

"如果我们用AI编程助手，开发效率能提升多少？"这个问题的答案和底层模型的SWE-bench表现直接相关。50%和60%之间的差距不是个数字，是你团队每天能自动化多少代码任务的实际能力边界。

**场景三：设计AI Native的产品功能**

你的产品是否需要内嵌代码生成能力？是否需要Agent自动修复Bug？理解评测设计能帮你设定合理的期望值，避免在产品规划里预设了模型还做不到的能力。

## 核心内容

### 第一代：HumanEval（2021）——已经没有区分度

**设计：** OpenAI在2021年推出HumanEval，包含164道Python函数编写题。给一个函数签名和文档字符串，让模型写出函数体，用单元测试验证正确性。

**现状：** 顶级模型pass@1已经超过90%，完全失去区分顶级模型的能力。

```
HumanEval pass@1 近年演化：
2021 Codex            ████████░░  72%
2023 GPT-4            █████████░  86%+
2024 Claude 3.5 S     █████████░  92%
2024 GPT-4o           █████████░  90%+
2024 Llama 3.1 405B   ████████░░  84%
```

**MBPP：** 大约500个Python问题，难度和HumanEval相近，现在同样基本饱和。这两个评测偶尔还会出现在发布材料里，但从产品决策角度来说，这些数字参考价值有限。

### 第二代：LiveCodeBench 和 BigCodeBench

**LiveCodeBench** 从LeetCode、Codeforces等竞赛平台持续抓取新题目，确保题目没有出现在任何模型的训练数据里。顶级模型在LiveCodeBench上的得分约50-65%（随难度分层变化），有效恢复了评测的区分度。

**BigCodeBench** 的设计更贴近真实开发：题目要求调用Python生态里的真实库（pandas、requests、PIL等），测试模型对API用法的掌握程度。

### 第三代：SWE-bench——仓库级别，最接近真实工作

SWE-bench（2023年，普林斯顿）是代码评测里里程碑式的设计突破。

**设计原理：** 从GitHub真实issue里抓取2294个已关闭的Bug报告（来自Django、Flask、NumPy等主流Python项目）。模型必须：
1. 理解issue描述（通常是自然语言的Bug报告）
2. 在整个代码仓库里定位相关代码
3. 生成补丁文件（Git diff格式）
4. 补丁要能通过原始项目的测试套件

```
SWE-bench Verified 解决率演化：
2023年底（早期模型）        ░░░░░░░░░░  约2-4%
2024年初（GPT-4+scaffold）  ░░░░░░░░░░  约10-16%
2024年中（Claude 3.5 S）    ░░░░░░░░░░  约49%
2024年底（Claude 3.7 S）    ░░░░░░░░░░  约62%
2025年初（顶级模型）         ░░░░░░░░░░  约55-65%
```

**SWE-bench Verified：** 原始SWE-bench有500个经过人工验证的子集，确保题目描述清晰、测试充分、答案明确。这是目前最常被引用的版本。

### 深入理解：SWE-bench 50% vs 60% 的实际差异

```
用实际工作类比：
假设开发团队每周有50个Bug需要修复：

模型A（50%通过率）：自动修复25个，剩25个需要人工
模型B（60%通过率）：自动修复30个，剩20个需要人工

这5个bug就是每周节省的工程师时间
按Bug修复平均2小时计算，每周节省10小时
对应约1.25个工程师工作日/周
年化约65个工作日，折合约0.25个FTE
```

更重要的是**哪类Bug多通过了**。60%通过率的模型在某些Bug类型上从完全不会变成了可以处理：理解长链路调用关系的Bug、需要理解框架约定的修复、错误信息描述模糊需要推理实际原因的Bug。

### 产品决策框架

```
SWE-bench Verified得分 → 适用产品形态（估算）

<20%    仅适合：代码补全、Snippet生成
20-40%  适合：单文件修改辅助、Code Review建议
40-60%  适合：简单Bug修复Agent、有监督的任务自动化
>60%    开始适合：半自动化Dev工作流、复杂Issue处理
>75%    （尚未实现）全面自动化Dev工作流
```

### Aider Polyglot 和跨语言评测

HumanEval和SWE-bench都以Python为主，但真实工程环境远比这复杂。

**Aider Polyglot** 在Python、Java、Go、TypeScript、Rust等多语言上测试代码编辑能力。顶模在Python上约65-70%，在Rust、Go上可能下降到50-60%，差距揭示了模型训练数据分布的偏向性。

## 能力边界与局限

**SWE-bench的局限一：只有Python，且主要是Web框架**

SWE-bench里的项目以Django、Flask、Pandas等Python生态为主。嵌入式系统、游戏引擎、前端React组件等场景基本没有覆盖。

**SWE-bench的局限二：不包含新功能开发**

SWE-bench测的是"修复已有Bug"，而工程师工作里很大一部分是"开发新功能"。修Bug和写新功能所需的能力有重叠但不等价。

**Agent框架的影响：**

同一个基础模型，配上不同的Agent框架（不同的工具调用策略、不同的上下文管理），SWE-bench得分可以相差10-20个百分点。这意味着**产品层实现**和底层模型一样重要。

---

## PM视角关键结论

1. **HumanEval/MBPP得分>90%就可以忽略，看SWE-bench**。对于代码能力选型，HumanEval已经没有区分度，SWE-bench Verified是目前最有参考价值的代码评测。

2. **SWE-bench 50%是当前AI代码Agent的实用门槛，60%是质量飞跃**。50%以下的模型很难支撑自主修Bug的产品场景；60%以上才开始具备半自动化开发工作流的能力基础。

3. **编程语言偏差是真实的产品风险**。如果你的工程团队主要用Go、Rust、TypeScript，要专门测这些语言的表现，不能用Python主导的SWE-bench分数代替。

4. **Agent框架层和模型层同等重要**。买/用哪个基础模型只是一半的故事，另一半是这个模型被包装成什么样的Agent（工具调用策略、上下文窗口管理、错误恢复机制）。

5. **LiveCodeBench的分数比HumanEval更可信**。月度更新的题目消除了数据污染，是评估模型代码能力时比较干净的信号。

## 常见误区

**误区一："代码评测高就说明模型适合当编程助手"**

评测测的是孤立任务的准确率，编程助手的体验还包括：响应速度、上下文窗口大小、多轮对话一致性、代码风格适配性、错误解释质量。一个SWE-bench得60%但响应慢、上下文短的模型，实际产品体验可能不如得55%但工程实现更优的竞品。

**误区二："SWE-bench 60%意味着能自动化60%的开发工作"**

SWE-bench题目是从已有项目抽取的历史Bug，难度分布和你的实际项目不同。你的项目可能有更复杂的依赖关系、更少的测试覆盖。实际自动化比例通常低于评测数字。

**误区三："开源模型代码能力接近闭源了，可以直接替换"**

某些开源模型在HumanEval上确实接近GPT-4o。但HumanEval已经饱和，这个接近没有意义。在SWE-bench这类更难的评测上，顶级闭源模型仍然有10-20个百分点的领先优势（截至2024-2025年），这个差距在工程实践中是显著的。

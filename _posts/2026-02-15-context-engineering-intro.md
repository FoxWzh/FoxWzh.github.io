**Context Engineering 不是 Prompt Engineering 的升级版，而是一个维度跃迁——从优化单次对话的措辞，到管理 AI 系统在整个任务生命周期中接收和处理的全部信息流。**

## 是什么

Context Engineering（上下文工程，简称 CE）是指对 AI 系统在推理过程中所接收的全部信息——包括系统指令、历史对话、检索内容、工具调用结果和当前用户输入——进行系统性设计、筛选、压缩和管理的工程实践。

如果说 Prompt Engineering 是在问"我应该怎么问这个问题"，那么 Context Engineering 在问的是"AI 在回答这个问题时，应该知道什么、不应该知道什么、知道多少"。

这个概念在 2024 年下半年开始受到广泛关注，直接触发因素是两个趋势的叠加：

1. **Agent 系统的爆发**：越来越多的 AI 产品不再是单轮问答，而是需要执行多步骤任务、调用多个工具、维持长期会话。在这种场景下，context 的构成远比"一个 prompt"复杂。

2. **上下文窗口的急剧扩大**：GPT-4 Turbo 支持 128K token，Claude 3 支持 200K token，Gemini 1.5 Pro 支持 100 万 token。理论上可以塞入更多信息，但"更多信息"不等于"更好的结果"，如何管理这个巨大的信息空间成为新的核心挑战。

## 为什么对 PM 重要

**场景一：RAG 产品的 context 设计**

某企业知识库 AI，底层使用向量检索 + GPT-4o。早期版本的问题：检索召回了 10 段相关文档，全部塞进 context，导致模型在 8000 token 的知识内容中"迷路"，回答质量反而比只检索 3 段时更差。

通过 CE 优化——对检索内容进行相关性排序、去重、压缩摘要，只保留最相关的 3-5 段，并在 context 中明确标注每段内容的来源和置信度——回答准确率从 61% 提升至 84%。

这不是 prompt 的问题，而是 context 组织方式的问题。

**场景二：代码助手的上下文管理**

GitHub Copilot、Cursor 等代码助手的核心竞争力不是底层模型，而是 context 工程能力——决定在用户输入的时候，应该把哪些文件、哪些代码片段、哪些 git 历史放进 context。

Cursor 的内部架构显示，它在每次代码补全时会动态构建一个 context 包，包含：当前文件的相关代码段、被引用的其他文件、近期编辑历史、项目级别的类型定义。这个动态构建过程就是 CE 的核心工作。

**场景三：客服 Bot 的会话记忆**

某金融 App 的 AI 客服，用户在第 1 轮说了账号名，第 3 轮报告了问题，第 5 轮询问解决方案。如果没有 CE，每轮对话的 context 只包含当前轮，模型在第 5 轮根本不知道用户是谁、遇到了什么问题。

通过 CE 设计——建立会话摘要机制，在每 5 轮对话后自动压缩生成摘要，将摘要注入后续轮次的 context——在保持 context 窗口合理使用的同时，维持了跨轮次的信息连贯性，用户满意度提升 31%。

## 核心内容

### PE 和 CE 的本质区别

```
Prompt Engineering 的视角：
┌──────────────────────────────┐
│  用户请求                     │
│    ↓                          │
│  [设计这个 prompt]            │  ← PE 关注的全部
│    ↓                          │
│  模型输出                     │
└──────────────────────────────┘

Context Engineering 的视角：
┌──────────────────────────────────────────────────────┐
│  信息来源层                                           │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐      │
│  │系统  │ │历史  │ │检索  │ │工具  │ │用户  │      │
│  │指令  │ │对话  │ │内容  │ │输出  │ │输入  │      │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘      │
│     │        │         │        │         │           │
│  ───┴────────┴─────────┴────────┴─────────┴───       │
│                    ↓                                  │
│            [CE 的核心工作区域]                        │
│    筛选 / 排序 / 压缩 / 结构化 / 动态注入             │
│                    ↓                                  │
│            Context Window（有限空间）                  │
│                    ↓                                  │
│                 模型推理                              │
└──────────────────────────────────────────────────────┘
```

**PE 优化的是"问什么"，CE 管理的是"知道什么"。**

两者不是替代关系，而是层次关系：好的 CE 是好的 PE 的前提。即使写了完美的 prompt，如果 context 中充满了噪声和无关信息，模型的输出质量也无法保证。

### Context 的五个组成部分

完整的 context 由以下五个部分构成，CE 的工作就是对这五个部分进行系统性管理：

**1. System Prompt**
产品方注入的角色定义、行为约束、输出格式。这部分相对静态，但在 Agent 系统中，可能随任务阶段动态切换。

**2. 历史对话（Conversation History）**
之前的 User/Assistant 轮次。在长对话场景下，全量保留历史会迅速消耗 context 窗口，需要摘要压缩机制。

**3. 检索内容（Retrieved Content）**
通过 RAG、向量搜索、数据库查询等方式获取的外部知识。这是最容易"过度塞入"的部分，也是 CE 设计中质量差异最大的部分。

**4. 工具调用结果（Tool Outputs）**
Agent 调用函数、API、代码执行后返回的结果。随着工具数量增加，这部分内容可能非常冗长，需要结果摘要和选择性注入。

**5. 当前用户输入（Current User Input）**
当前轮次的用户消息。看似简单，但在复杂场景下（用户上传文件、图片等），这部分的 CE 设计也很重要。

### Context 的价值密度

Context Engineering 的核心指标不是"塞了多少内容"，而是"每个 token 携带了多少有效信息"，这就是**上下文价值密度（Context Value Density）**。

```
低价值密度 context（每 100 token 有效信息量 = 20）：
┌────────────────────────────────────────────────┐
│ 这是一份关于产品的文档。产品名称是 XX。这个产品  │
│ 有很多功能。功能包括 A 功能，A 功能可以帮助用户  │
│ 做 A 相关的事情。同时还有 B 功能，B 功能也很重要│
│ ...（重复、冗余、无结构）                        │
└────────────────────────────────────────────────┘

高价值密度 context（每 100 token 有效信息量 = 80）：
┌────────────────────────────────────────────────┐
│ 产品：XX Pro | 版本：3.2 | 发布：2025-03        │
│ 核心功能：A（自动化报表）B（多维分析）C（告警）  │
│ 定价：Basic ¥299/月 | Pro ¥899/月              │
│ 用户问题高频 Top3：登录失败 / 数据导出 / 权限    │
└────────────────────────────────────────────────┘
```

提升价值密度的方法：
- 结构化信息（表格、列表）比段落文本更高效
- 去除修饰性语言，保留事实核心
- 对检索结果进行相关性过滤，宁可少而精
- 使用摘要替代全文，保留关键点

### 主动 context 管理 vs 被动 context 填充

大多数早期 AI 产品采用的是**被动 context 填充**策略：用户说什么就传什么，检索到什么就塞什么，对话有多长就带多长。这种方式简单，但随着系统复杂度增加，会导致性能和质量双重下降。

成熟的 AI 产品应该实施**主动 context 管理**：

| 管理维度 | 被动填充 | 主动管理 |
|---------|---------|---------|
| 历史对话 | 全量传入 | 滑动窗口 + 摘要压缩 |
| 检索结果 | Top-K 全部传入 | 相关性评分 + 选择性传入 |
| 工具结果 | 完整结果 | 关键字段提取 + 摘要 |
| Context 大小 | 无上限控制 | 预算管理（如最多 4K token 用于检索内容）|
| 内容更新 | 静态（每轮不变）| 动态（根据对话进展调整注入内容）|

### 为什么 CE 在 Agent 时代变得关键

在单轮问答场景中，context 相对简单：系统提示 + 用户问题。PE 足以处理这个场景。

但在 Agent 场景中，context 的复杂度呈指数增长：

```
单轮 Q&A 的 context 构成：
[System] + [User Question]  →  简单，PE 足够

Multi-step Agent 的 context 构成：
[System] + [Task Goal]
    + [Step 1 Plan]
    + [Tool Call 1 Result（可能很长）]
    + [Step 2 Plan]
    + [Tool Call 2 Result]
    + [Retrieved Knowledge]
    + [Previous Conversation Summary]
    + [Current User Instruction]
    →  复杂，必须 CE
```

一个运行 10 步工具调用的 Agent，如果每步工具返回 500 token 的结果，仅工具输出就占用了 5000 token。加上其他 context 组成部分，极易超出上下文窗口或严重影响推理质量。

CE 在 Agent 场景中的具体工作包括：
- **工具结果摘要**：将 API 返回的 JSON 数据提炼为关键摘要
- **跨步骤状态管理**：维护 Agent 在多步骤任务中的"工作记忆"
- **动态上下文剪枝**：当 context 接近窗口上限时，自动压缩早期步骤的信息
- **任务感知检索**：根据当前任务阶段，动态决定检索什么知识注入

### PE 到 CE 的演化时间线

```
2020 ──────────────────────────────────────────────── 2025

  │    GPT-3         GPT-4         Claude 3     Agent 爆发
  │    发布          发布          发布         时代
  │      │             │             │             │
  ▼      ▼             ▼             ▼             ▼

  Zero-shot   Few-shot     CoT/System    CE 成为
  PE 探索     PE 成熟      Prompt 精细化  核心工程能力

触发条件：
  ┌─────────────────────────────────────────────────┐
  │ 上下文窗口：4K → 128K → 200K → 1M token        │

  单次 prompt 的质量     整个信息流的管理
  ← PE 的边界         CE 的起点 →

```

## 三个真实场景的 CE 设计分析

**场景一：RAG 产品的 context 设计**

关键设计决策：检索多少文档（K 值）？如何排序？是否需要二次重排（Reranking）？

优秀 RAG 产品的 CE 实践：
- 检索 20 个候选，用 Reranking 模型筛选出最相关的 5 个
- 每个检索段落附带来源元数据（文档名、更新时间、章节）
- 相关性低于阈值的段落不注入，即使检索到也丢弃
- 检索内容 token 预算硬上限（如 3000 token），防止挤占推理空间

**场景二：代码助手的上下文管理**

关键 CE 挑战：当用户修改一个函数时，应该把哪些代码文件/片段放进 context？

Cursor 等工具的 CE 策略：
- 当前文件完整内容（高优先级）
- 被当前文件 import 的关键依赖（中优先级）
- 最近 5 次编辑的文件（低优先级）
- 项目 README 和 tsconfig/package.json（背景知识）
- 总 context 预算：不超过 32K token，优先级低的截断

**场景三：客服 Bot 的会话记忆**

关键 CE 挑战：长对话场景下，如何在 context 窗口内维持信息连贯性？

有效 CE 策略：
- 每 5 轮生成一次对话摘要（用较小的模型，降低成本）
- 摘要包含：用户身份、已确认的问题、已尝试的解决方案
- 后续轮次只传入：摘要 + 最近 3 轮原始对话 + 当前输入
- 关键实体（订单号、账号）单独提取存储，每轮自动注入

## 能力边界与局限

1. **CE 无法突破上下文窗口的物理限制**。即使是 1M token 的窗口，在成本和延迟约束下，也有实际的有效信息上限。
2. **CE 的效果依赖对业务场景的深度理解**。没有通用的 CE 方案，不同产品场景需要定制化的 context 管理策略。
3. **CE 增加了系统复杂度**。引入摘要、重排、动态注入等机制，会显著增加工程复杂度和延迟，需要在效果和成本之间仔细权衡。
4. **评估 CE 效果很难**。不像 PE 可以用单次 prompt 测试对比，CE 的效果往往需要在完整的多轮、多步骤场景下才能体现，评估成本更高。

## PM 视角关键结论

1. **CE 是 AI 产品工程能力的核心竞争力**。当底层模型能力趋于同质化（GPT-4 级别的模型越来越多），context 管理质量成为差异化的主要来源。同样的底层模型，CE 做得好的产品体验可以比 CE 差的产品好 2-3 倍。

2. **Context 管理质量直接决定产品体验天花板**。再好的模型，如果 context 里充满噪声和无关信息，输出质量就会受限。CE 不是优化项，而是基础能力。

3. **建立 Context 预算的概念**。像管理内存一样管理 context——为每个 context 组成部分设置 token 预算上限，确保高价值信息不被低价值信息挤出。这是产品决策，不只是工程参数。

4. **从产品设计阶段就考虑 CE**。CE 不是事后修补，而是应该在产品架构设计阶段就纳入考虑。"AI 在执行这个任务时需要知道什么"应该是产品 PRD 的基本要素。

5. **CE 能力是 AI 产品技术壁垒的核心**。外部竞争者可以用同样的底层 API，但你积累的 context 管理知识（什么信息重要、什么信息应该过滤、如何组织信息结构）是难以复制的护城河。

## 常见误区

**误区一：上下文窗口越大，context 管理越不重要**

很多团队认为 Gemini 1.5 支持 100 万 token 就可以把所有相关信息全部塞进去，不需要精细的 CE。实验数据一再表明，"大海捞针"式的超长 context 在实际推理中效果不如"精准注入"的短 context。模型的注意力机制在超长 context 下存在明显的性能衰减，而且成本也会随 context 长度线性增加。

**误区二：CE 是工程师的工作，PM 不需要了解**

CE 的核心决策——应该给 AI 哪些信息、不给哪些信息、如何组织——本质上是产品决策，需要对业务场景和用户需求有深度理解，这恰恰是 PM 的优势所在。把 CE 完全交给工程师，往往导致 AI 系统的信息架构脱离实际业务需求。

**误区三：CE 只在技术层面起作用，对用户体验无直接影响**

CE 的影响直接体现在用户体验上：AI 是否记住了上轮说过的事情？是否给出了与当前问题真正相关的答案？是否在多步任务中保持了逻辑一致性？这些都是用户可以直接感知的体验差异，根源都在 CE。

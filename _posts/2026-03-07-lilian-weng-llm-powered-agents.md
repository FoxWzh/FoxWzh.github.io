## 原文核心内容

### 一张图看懂 Agent 系统

原文开篇给出的框架极其简洁：LLM 是大脑，Agent 系统由三个模块围绕它构建：

```
         ┌──────────────────────────────────────────┐
         │           LLM（核心控制器）                │
         └──────────────┬───────────────────────────┘
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
    规划（Planning） 记忆（Memory） 工具（Tool Use）
    · 任务分解      · 短期记忆      · 外部 API 调用
    · 自我反思      · 长期记忆      · 计算器、搜索
```

这个三模块框架在 2023 年是第一次被如此清晰地系统化，影响了后来几乎所有的 Agent 框架设计。

---

### 模块一：规划（Planning）

规划模块解决的问题是：面对一个复杂任务，Agent 如何知道该做什么、按什么顺序做、做错了怎么纠正。

#### 任务分解

**Chain of Thought（CoT，Wei et al. 2022）**

让模型"逐步思考"，把大任务分解成小步骤。这已经是标配的提示技术，但原理值得理解：CoT 本质上是在用测试时的算力（test-time compute）换准确率——让模型多"想"几步，而不是直接给答案。

**Tree of Thoughts（ToT，Yao et al. 2023）**

CoT 是一条线，ToT 变成了一棵树。每一步不只生成一个思路，而是生成多个，然后用 BFS 或 DFS 搜索最优路径，每个分支用分类器或提示词打分。适合有明确最优解的推理问题，但计算成本高。

**LLM+P（Liu et al. 2023）**

把规划这一步直接外包给经典规划器（PDDL）：LLM 先把问题翻译成规划语言，交给专业规划器生成计划，再把计划翻译回自然语言。这种"专业工具做专业事"的思路在机器人领域很实用，但通用场景里 PDDL 格式化成本高。

#### 自我反思

这是原文里最有价值的部分之一，讲的是 Agent 如何从错误中学习。

**ReAct（Yao et al. 2023）**

把推理（Reasoning）和行动（Acting）交织在一起。每次行动前先"想"，想完再"做"，做完观察结果再"想"——形成 Thought → Action → Observation 的循环。

```
Thought: 我需要知道法国的首都
Action: 搜索[法国首都]
Observation: 巴黎是法国的首都
Thought: 找到了，可以回答了
Action: 完成[答案：巴黎]
```

ReAct 的核心价值在于：行动和推理不分离，模型在做之前先想，做完之后再想。实验表明 ReAct 比纯 CoT 或纯行动策略都更好，且幻觉更少。

**Reflexion（Shinn et al. 2023）**

在 ReAct 基础上加了一层"事后反思"。Agent 完成一次尝试后，会对自己的行为进行内省，生成一段自我批评，这段批评作为记忆保存下来，影响下一次尝试。

内省的提示词长这样：
> "你是一个能够基于自我反思改进的高级推理 Agent。你将获得一次之前的尝试，你的目标是……写一段反思，描述哪里出了问题，以及你下次尝试会如何改进。"

Reflexion 能帮 Agent 识别幻觉、纠正低效规划、减少重复行为。本质上是给 Agent 加了一个"复盘"机制。

**Chain of Hindsight（CoH，Liu et al. 2023）**

让模型从一系列带反馈的历史输出中学习。训练数据格式是 `(prompt, output_1, feedback_1, output_2, feedback_2, ...)`，模型学习的不是"正确答案"，而是"如何从旧输出改进到新输出"。相当于把人类迭代改稿的过程编码进模型的训练里。

**Algorithm Distillation（AD，Laskin et al. 2023）**

把 CoH 的思路用到强化学习里：把 Agent 在多个 episode 里的学习历史作为上下文输入，让模型直接预测"下一步会更好的行动"。学的不是某个静态策略，而是"如何学习"这个过程本身。

### 模块二：记忆（Memory）

原文用人类记忆系统作类比，把 Agent 的记忆分三层：

| 人类记忆 | 时长 | 容量 | 对应 LLM |
|---------|------|------|---------|
| 感官记忆（Sensory） | 几秒 | 极小 | 原始输入 / Embedding |
| 短期记忆（Working） | 20-30秒，约7个组块 | 小 | Context Window |
| 长期记忆（Long-term） | 数天到数十年 | 近似无限 | 外部向量数据库 |

这个类比很有价值：context window 就是 Agent 的工作记忆，撑不过一个 session，而且容量有限；真正的"记住"需要外部向量存储。

#### 长期记忆的实现：向量检索

外部记忆的读取靠 **MIPS（Maximum Inner Product Search）**，即找最相似的向量。常见算法：

- **LSH（局部敏感哈希）**：把相似向量哈希到同一个桶，速度快但精度有损
- **ANNOY**：随机投影树，适合静态数据集
- **HNSW（分层可导航小世界）**：建层级图，底层存数据点，上层是"快捷方式"，速度和精度都很好，是目前最常用的方案之一
- **FAISS（Facebook AI Similarity Search）**：假设高维空间存在聚类结构，用向量量化压缩检索
- **ScaNN（Google）**：各向异性向量量化，专为最大内积搜索优化

实际产品里用 HNSW 或 FAISS 最多，Pinecone、Weaviate、Chroma 等向量数据库底层基本都是这些算法。

### 模块三：工具调用（Tool Use）

工具调用让 LLM 突破训练截止日期和知识边界——模型权重是固化的，但通过调用 API，可以访问实时数据、执行代码、查专有数据库。

**MRKL（Karpas et al. 2022）**

神经符号架构：LLM 作为路由器，把任务分发给一组"专家模块"（可以是其他神经网络，也可以是符号系统比如计算器、货币转换器、SQL 引擎）。这是 Tool Use 最早的系统化论文之一。

**Toolformer（Schick et al. 2023）**

微调 LM 让它自己学会调用 API——先生成可能有用的 API 调用，看调用结果是否改善了输出，如果改善了就保留这次调用。这样模型能在没有人工标注的情况下自己学会什么时候调什么工具。

**HuggingGPT（Shen et al. 2023）**

用 ChatGPT 作为任务规划器，根据 HuggingFace 上的模型描述来分配任务给专门的模型。四个阶段：
1. 任务规划：LLM 把用户请求解析成任务列表
2. 模型选择：LLM 根据描述把任务分配给最合适的专家模型
3. 任务执行：专家模型执行并返回结果
4. 响应整合：LLM 把结果汇总成最终答案

本质上是 LLM 作为 orchestrator，其他专门模型作为 worker 的多模型协同架构。

**API-Bank（Li et al. 2023）**

包含 53 个常用 API 的工具调用评测基准，三个难度等级：
- Level 1：会调某个 API（给你 API 文档，知道要不要调、怎么调）
- Level 2：会检索 API（从一堆 API 里找到该用哪个）
- Level 3：会规划多步 API 调用（复杂请求需要串联多个 API）

### 案例：Generative Agents（Park et al. 2023）

原文用最多篇幅介绍的案例。25 个虚拟角色各由一个 LLM-powered Agent 驱动，在类《模拟人生》的沙盒环境里生活和互动，产生了令人惊讶的涌现行为——角色会自发组织聚会，会传递信息，会形成社交关系。

架构的核心是 Memory Stream + Retrieval：
- **Memory Stream**：用自然语言记录 Agent 的完整经历，每条记忆附带时间戳
- **Retrieval Model**：根据三个维度打分——新近性（recency）、重要性（importance）、相关性（relevance），加权后检索最相关的记忆
- **Reflection Mechanism**：定期把低层记忆合成为高层洞见（"Isabella 喜欢浪漫的活动" 这类总结性记忆）
- **Planning & Reacting**：把反思和环境信息转化成行动计划

这个案例展示了一个关键结论：给 Agent 加上"如何管理自己的记忆"的机制，比单纯加大 context window 更能产生连贯的、人性化的行为。

### 当前挑战

原文最后列了三个核心挑战，2023 年提出，今天依然成立：

**1. Context 长度有限**

历史信息、详细指令、复杂 API schema 都要挤进有限的 context window。向量检索能部分缓解，但检索出来的片段无法替代全文注意力的理解深度。

**2. 长期规划难**

在一个很长的行动序列里保持目标一致性，遇到意外错误时灵活调整计划——这是现在的 LLM 最薄弱的地方之一。很容易陷入局部优化，或者在多步骤后偏离原始目标。

**3. 自然语言接口不可靠**

LLM 和外部工具之间的接口是自然语言（或 JSON），格式错误、指令被忽略、工具输出解析失败——这些"接口可靠性"问题在实际部署中是高频踩坑点。

## 我的解读

### 为什么这篇文章在 2026 年还值得读

原文发布于 2023 年 6 月，距今快三年。LLM 能力已经翻天覆地，但这篇文章仍然有价值，原因不是它讲的技术有多新，而是**它提供了一套思考 Agent 系统的语言和框架**。

规划-记忆-工具这三个维度，今天依然是设计 Agent 系统时最自然的拆解方式。Anthropic 的 Claude Agents、OpenAI 的 Assistants API、LangGraph 的工作流设计，背后的思维框架都没有跳出这个三角形。

### 三个值得拿走的具体洞见

**1. ReAct 的价值不只是提示词技巧**

很多人把 ReAct 当成一种 prompt 格式：Thought/Action/Observation 循环。但原文强调的是它的本质——把推理和行动交织，而不是"先计划再执行"的线性分离。

这对产品设计的启发：不要设计"先让 AI 出完整计划，再一次性执行"的 Agent，而要设计"每步行动前想一想，行动后观察再想"的循环。前者在面对意外时容易崩，后者更有弹性。

**2. Reflexion 的本质是给 Agent 加复盘机制**

Reflexion 在工程上的价值被低估了。它做的事情很简单：把上次失败的经验作为上下文传给下一次尝试。这不需要微调，只需要把历史反思存起来、塞进 prompt。

对于构建 Agent 产品的团队：在 Agent 失败后，不要只是"重试"，要让 Agent 先"复盘"再重试。这是 Reflexion 教的。

**3. 记忆管理是 Agent 产品最被低估的基础设施**

Generative Agents 的案例说明：Agent 的行为质量，很大程度上取决于它**如何管理自己的记忆**——什么记住、什么遗忘、什么时候检索。这比换一个更强的底层模型更能影响行为的连贯性和"人性"。

对于在做 Agent 产品的 PM：记忆系统的设计（存什么、怎么检索、多久合成一次高层记忆）应该和模型选型同等优先级，而不是事后才想到的"持久化功能"。

### 原文挑战的现状更新（2026 年视角）

原文提出的三个挑战，三年后进展如何：

**Context 长度**：大幅缓解但未解决。Claude 3.7 有 200K，Gemini 1.5 Pro 有 1M。但"有 context"和"有效利用 context"之间仍有差距——长文档里的中间部分仍然容易被忽略（Lost in the Middle 问题）。

**长期规划**：依然是最薄弱的地方。o1/o3 通过 CoT 在推理任务上有显著提升，但在多步骤的现实任务执行中，Agent 仍然容易偏轨。SWE-bench 的顶模表现（60% 出头）部分反映了这个限制。

**接口可靠性**：随着 Function Calling / Tool Use 的标准化有实质改善。Claude 3.7 在 BFCL 上达到 ~91%，JSON 格式错误率大幅降低。但在复杂 schema 和中文工具描述场景下仍有问题。

## 延伸思考

**从 2023 到 2026：Agent 从概念到产品**

原文写于 AutoGPT 和 BabyAGI 刚刚爆火的时候，那时候 Agent 更多是 demo——能跑，但不能用于生产。

三年后，Agent 进入了产品化阶段：Cursor/Windsurf 的代码 Agent、Devin 的工程 Agent、Claude Projects 和 OpenAI Assistants 的记忆能力……这些产品都是在原文的框架里一块一块填空。

哪些块还没填好：长期规划（规划模块）、多步骤任务的可靠性（工具模块的容错）、跨 session 的连贯记忆（记忆模块的持久化）。

**Agent 的"可靠性"才是真正的产品问题**

原文提的三个挑战，说到底是一个问题：**Agent 在现实任务里还不够可靠**。

这和模型能力是两回事。一个能力很强的模型，因为接口格式错误、因为上下文管理不当、因为规划偏轨，最终也会在用户面前失败。

做 Agent 产品，核心工程问题不是"怎么用最强的模型"，而是"怎么让整个系统的失败率降到用户可以接受的范围"。这需要规划模块加 Reflexion 机制，记忆系统加检索质量评估，工具调用加容错和重试逻辑。

这是原文在 2023 年就已经看清楚、但工业界花了很长时间才真正当成第一优先级的问题。

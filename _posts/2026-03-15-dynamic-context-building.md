**静态prompt是AI产品的"幼稚园"阶段——真正有竞争力的AI产品，依靠动态context构建实现千人千面的智能信息装配。**

---

## 是什么

大多数AI产品的早期形态是这样的：工程师写一段固定的system prompt，每次用户发起对话时，这段prompt原封不动地发给模型。这就是静态context。

静态context有它的价值：简单、可预测、易调试。但它有一个根本性的局限：它对所有用户、所有场景、所有时间点一视同仁。一个不知道你是谁、不知道你现在在哪个页面、不知道你最近买了什么的AI，只能给出通用答案。

动态context构建（Dynamic Context Building）是指：在每次模型调用前，根据当前用户、当前任务、当前环境，实时从多个来源收集信息，智能组装成最适合这次调用的context。它不是一段固定的文本，而是一个实时运行的装配过程。

---

## 为什么重要：三个产品场景

**场景一：医疗问诊AI的个性化**
某互联网医疗平台的AI助手，静态版本对所有用户使用相同的prompt，导致一位已确诊高血压的老年用户咨询感冒药时，AI推荐了含伪麻黄碱的药物（对高血压患者有风险）。动态context版本在每次对话前将用户的既往病史、过敏史、当前用药注入context，彻底避免了这类风险。

**场景二：代码助手的项目感知**
GitHub Copilot的进化历程展示了动态context的价值：早期版本只看当前文件，后期引入了"repomap"机制，动态分析整个代码仓库的结构、相关文件、项目约定，补全质量大幅提升。

**场景三：电商客服的实时装配**
某电商平台的AI客服，在用户发起对话的瞬间，动态检索该用户最近3个未完成订单、最近浏览的商品、账户余额状态，装配进context。这使得AI可以主动说"我看到您有一个昨天下单的商品还在配送中"，而非等用户解释订单号。

## 静态 vs 动态：本质区别

```
静态Context（固定模式）：

[用户发起对话]
      ↓
[固定System Prompt] + [用户输入]
      ↓
[模型生成回复]

动态Context（装配模式）：

[用户发起对话]
      ↓
[触发Context装配引擎]
   ↙    ↓    ↘    ↘
[RAG]  [工具]  [画像] [历史]
   ↘    ↓    ↙    ↙
[优先级排序 + 裁剪]
      ↓
[System Prompt] + [装配好的Context] + [用户输入]
      ↓
[模型生成回复]
```

## 动态Context的四个来源

### 来源一：实时检索（RAG）

根据用户的当前查询，从知识库中检索相关内容动态注入。

关键设计决策：
- **检索时机：** 在用户输入后、模型调用前触发检索（而非预先全量注入）
- **检索粒度：** 段落级（通常300-500字一段）优于文档级
- **相关性阈值：** 设定最低相似度分数，过滤低质量检索结果
- **数量上限：** 通常3-7段，避免稀释效应

### 来源二：工具调用结果

调用外部API获取实时数据后，将结果格式化后注入context。

常见工具：
- 数据库查询（用户账户、订单状态）
- 外部API（实时价格、库存、天气）
- 代码执行结果
- 浏览器/搜索结果

工具结果注入的关键原则：**先压缩，再注入**。工具返回的原始数据往往冗余，直接注入会造成稀释。应提取关键字段，结构化后再放入context。

### 来源三：用户画像

持久化的用户偏好、历史行为、身份信息。

分层管理：
```
强属性（高置信度）：
- 明确设置的偏好（语言偏好、专业背景）
- 已验证的事实（年龄、会员等级）
→ 直接注入，无需检索

软属性（推断的）：
- 行为推断的偏好（常购品类、阅读习惯）
- 近期意图信号
→ 按相关性选择性注入
```

### 来源四：历史摘要

将历史对话压缩为结构化摘要，保留关键信息。

与完整历史对话的区别：
- 完整历史：原文保留，占用大量tokens
- 历史摘要：提炼关键事实，压缩比10:1或更高
- 历史摘要注重"已建立的共识"而非"对话过程"

## Context装配的优先级机制

当所有来源的信息加起来超出context窗口时，需要一套明确的取舍规则：

```
优先级框架（由高到低）：

P0 - 永不删除（约5-10% context空间）：
  ├── 核心系统指令
  ├── 安全约束
  └── 用户关键身份信息

P1 - 优先保留（约20-30% context空间）：
  ├── 当前轮用户输入
  ├── 最近2轮完整对话
  └── 与当前查询强相关的检索结果

P2 - 按需保留（约40-50% context空间）：
  ├── 与当前查询中等相关的检索结果
  ├── 用户画像中的相关软属性
  └── 近期历史摘要

P3 - 优先裁剪（约10-20% context空间缓冲）：
  ├── 低相关性检索结果
  ├── 历史对话中的闲聊轮次
  └── 已完成任务的工具调用结果
```

当context接近上限，从P3开始裁剪，保持P0内容始终完整。

## 相关性评分：判断哪些检索结果值得放入

相关性评分是动态context构建的核心技术。主要有三种方法：

**方法一：向量相似度（最常用）**
将用户查询和知识库段落都转换为向量，用余弦相似度判断相关性。简单高效，但对语义跳跃（用间接方式描述需求）敏感度不足。

**方法二：交叉编码器重排（Cross-Encoder Reranking）**
先用向量检索获取Top 20候选，再用交叉编码器模型精排取Top 5。效果更好，但增加了一次模型调用的延迟（通常100-300ms）。

**方法三：LLM判断（最准确但成本最高）**
让一个轻量级LLM判断"这段内容是否有助于回答用户问题"。准确性最高，通常只用于高价值场景（如医疗、法律）。

对于大多数AI产品，推荐方案：向量检索 + 规则过滤（最低阈值0.75）+ 数量上限（5段）。在质量要求更高的场景引入交叉编码器重排。

## 多Agent场景下的Context隔离 vs 共享

在多Agent架构中，context管理面临一个新的设计问题：Agent之间的信息应该隔离还是共享？

```
隔离设计（适用于并行独立任务）：

Agent A           Agent B           Agent C
[Context A]       [Context B]       [Context C]
   ↓                 ↓                 ↓
[输出A]           [输出B]           [输出C]
         ↘           ↓          ↙
              [汇总Agent]
          [Context: 仅包含各Agent输出]

优势：防止Agent间相互污染；便于并行执行
适用：并行处理不同文档/任务的场景
```

```
共享设计（适用于协作型任务）：

           [共享Context Pool]
          /         |          \
Agent A            Agent B            Agent C
（读写权限）      （只读权限）        （读写权限）
   ↓                ↓                   ↓
[基于全局状态的输出]

优势：Agent间信息协同；避免重复工作
风险：一个Agent的错误污染全局状态
适用：需要状态共享的顺序型协作任务
```

设计原则：**默认隔离，按需共享**。只有当Agent间确实需要信息流动时，才设计共享机制，并添加写入验证层防止污染。

## 真实案例：电商导购AI的动态Context装配

以一个典型的电商导购AI为例，展示完整的动态context构建流程：

**用户行为：** 某用户在家电分类页面停留了5分钟后，进入AI对话框输入"我想买一台适合小户型的空调"

**Context装配过程：**

```
Step 1：解析用户意图
→ 意图识别：购买咨询 + 产品品类=空调 + 约束=小户型

Step 2：并行信息收集（约200-400ms）
┌────────────────────────────────┐
│ 线程A：RAG检索                  │
│ 查询："小户型空调推荐"           │
│ 结果：Top 5产品详情+用户评价摘要 │
└────────────────────────────────┘
┌────────────────────────────────┐
│ 线程B：用户画像读取              │
│ 读取：地区=上海、历史订单、       │
│ 收藏夹商品、会员等级=黄金会员    │
└────────────────────────────────┘
┌────────────────────────────────┐
│ 线程C：实时工具调用              │
│ 查询：用户收藏的2台空调当前库存  │
│ 查询：黄金会员当期优惠券         │
└────────────────────────────────┘

Step 3：装配 + 优先级排序
[系统角色定义] → P0
[用户画像关键信息：上海、黄金会员] → P1
[用户收藏商品库存+优惠券] → P1
[RAG: Top 3相关空调产品信息] → P2
[RAG: 小户型空调选购指南摘要] → P2
[用户历史购买品类偏好] → P3

Step 4：模型调用
AI回复："我看到您收藏了X品牌1匹和1.5匹两款，
考虑到上海的气候特点，小户型通常推荐1匹..."
```

这个过程在用户感知层面不超过1秒，但在背后完成了多源数据的并行收集和智能装配。

## 能力边界与局限

**装配延迟是不可避免的代价：** 动态context构建需要并行调用多个数据源，即使优化到极致，也会带来200-500ms的额外延迟。对于实时性要求极高（<200ms响应）的场景，需要在动态程度和速度之间做出权衡。

**装配逻辑的调试难度：** 静态prompt出错很容易定位，但动态context装配的问题可能来自任何一个数据源或装配逻辑。这要求配套建立完善的可观测性工具（每次调用的context快照、各来源的贡献记录）。

**相关性判断的不确定性：** 没有任何检索系统能100%准确判断什么信息对当前查询有用。对于模糊或多义的用户输入，检索结果的质量会显著下降，影响最终回答。

**冷启动问题：** 对新用户，没有画像、没有历史，动态context退化为半静态context。需要设计新用户的冷启动策略（如主动询问关键信息、使用群体画像作为默认值）。

## PM视角的五条关键结论

**1. 动态Context是AI产品的核心技术壁垒，很难被简单复制**
一个好的动态context装配系统，需要精心设计的信息架构、优化过的检索管道和持续迭代的优先级策略。这些积累形成了竞争护城河，比单纯的prompt设计更难被竞争对手快速复制。

**2. 好的检索 + 好的装配 > 更好的模型**
在很多场景下，将模型从GPT-4 mini升级到GPT-4 Turbo带来的质量提升，远小于将检索精度从F1=0.6提升到F1=0.8带来的提升。PM应该推动在"信息装配质量"上的投入，而非只盯着模型版本。

**3. Context装配逻辑要可观测、可调试，这是产品质量保证的基础**
必须能够回答"这次对话中模型看到了什么"这个问题。每次模型调用的context快照应该被记录，作为质量排查和优化的依据。没有可观测性的黑盒装配是不可靠的。

**4. 并行收集信息，优先级裁剪，而非顺序收集**
动态context构建的工程优化关键：让各数据源的读取并行执行，在汇总阶段做优先级裁剪。如果顺序执行多个数据源读取，延迟会随数据源数量线性增长，用户体验难以接受。

**5. 新用户冷启动是被低估的产品挑战**
大多数团队在设计动态context系统时，默认已有丰富的用户数据。但新用户第一次使用时，动态context能装配的信息非常有限，体验往往最差——而第一印象对留存率影响最大。冷启动策略应该作为独立的产品设计问题来处理。

## 常见误区

**误区1："有了RAG就等于有了动态context"**
RAG只是动态context四个来源之一，而且只覆盖了"知识"这一维度。没有用户画像、没有实时工具调用、没有历史摘要，即使有RAG，仍然缺少用户个性化和实时状态感知。完整的动态context是四个来源的协同。

**误区2："动态context越丰富越好，信息越多越准"**
信息量和回答质量并非线性关系。装配进过多信息（特别是低相关性信息）会触发稀释效应，反而降低模型对关键指令的遵循。动态context的目标是"最相关的信息"，而非"最多的信息"。

**误区3："context装配是一次性设计，上线后不需要迭代"**
动态context装配系统需要持续优化：随着用户行为变化调整画像权重，随着知识库增长优化检索策略，随着新功能上线增加新数据源。把它作为一个需要持续运营的系统，而非一次性建好就可以的基础设施。
